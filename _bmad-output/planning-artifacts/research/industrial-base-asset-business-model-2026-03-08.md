## 1. 结论速览

### 1.1 核心判断

1. 当前公司并不是“若干独立项目系统”的松散组合，而是已经形成了一个以 `OLTran` 为品牌底座、以 `CDM/TDM/管理信息化` 为三条主业务线的产品雏形。  
   其中：
   - `OLTran.CDM` 对应协同研发平台。
   - `OLTran.TDM` 对应数字化试验平台。
   - `OLTran.SUNBPM` 对应管理信息化/科研业务管理平台。
   - `OLTran.SIM / OLTran.iDesigner / OLTran.VLab` 属于平台向仿真、工业 APP、虚拟试验方向的延展产品族。  
   这是本次研究最重要的资产识别结果之一。[E1][E2][E3][E4]

2. 自有产品线已经隐含出一套“底座共性能力”：
   - 统一对象/元模型
   - 数据引擎、流程引擎、协同引擎、工具引擎
   - 项目/任务/需求/数据/知识中心
   - 主数据、权限、文件服务、消息、搜索、图谱、规则
   - IT/OT 双向集成
   - 微服务化、混合存储、高可用部署  
   换言之，真正需要被架构化和治理的对象，不是某个单一产品，而是这套共性能力。[E1][E2][E3][E16]

3. 竞品市场大致分成五种范式：
   - `数据模型/数字主线范式`：华为 DME、易立德、杰为
   - `流程任务一体化协同研发范式`：索为
   - `仿真流程与数据管理范式`：安世亚太
   - `通用低代码 BPM/PaaS 范式`：炎黄盈动
   - `集成展示/部署导向平台范式`：天舟上元（基于截图，证据强度较弱）  
   这意味着你后续的底座设计不能只做成“低代码平台”，也不能只做成“PDM/PLM 替代品”，而应明确瞄准“工业研发底座”。[E9][E10][E11][E12][E13][E14][E15]

4. 对底座设计最值得坚持的总方向是：
   - 对内：打通 `协同研发 + 数字化试验 + 管理信息化`
   - 对外：建立 `统一对象模型 + 统一集成面 + 统一治理规则`
   - 对业务：围绕 `需求 -> 项目 -> 工作包/任务 -> 设计/仿真/试验 -> 数据 -> 知识 -> 管理决策`
   - 对技术：沉淀成可复用、可配置、可扩展、可治理的“工业研发共性平台”

### 1.2 本次资料可用性判断

- `docs/reference` 共计 `182` 个文件。
- 其中 `23` 个文件为空文件，全部位于 `2-管理信息化` 目录。
- `00-竞争对手` 共 `140` 个文件，其中图片 `86` 个，已通过联系图筛选和关键图多模态解读补充。

- 因此：
  - `协同研发`、`TDM`、`竞品主线` 的结论可信度较高。
  - `管理信息化` 的功能边界可信度中等，但其产品组合轮廓已经足够清晰。

## 4. 业务建模

## 4.1 一级业务域划分

建议将现有资产抽象为以下 `4+1` 个一级业务域：

1. **研发协同域**
   - 负责需求、项目、工作包、任务、评审、变更、交付物、研发门户。
   - 对应 `CDM` 主线。

2. **试验数据域**
   - 负责试验计划、采集、传输、存储、建模、分析、可视化、试验站点、虚实融合。
   - 对应 `TDM` 主线。

3. **管理信息化域**
   - 负责科研计划、科研项目、合同、设备器材、主数据、审批与经营管理。
   - 对应 `SUNBPM/科研业务管理系统` 主线。

4. **平台共性域**
   - 负责身份、组织、角色、权限、字典、元模型、文件、搜索、消息、规则、流程、连接器、运维。
   - 是底座设计的真正“平台层”。

5. **知识与智能域**
   - 横切三条业务线，负责知识库、知识图谱、知识推荐、语义检索、大模型辅助。
   - 建议做成独立平台服务，而不是绑死在单一产品。

## 4.2 核心业务对象模型

| 对象 | 所属域 | 关键关系 | 架构意义 |
|---|---|---|---|
| `Requirement` 需求/指标 | 研发协同域 | 驱动项目、工作包、验证、知识、试验 | 是研发链条起点，应成为统一标识对象 |
| `Project / Plan / WBS / WorkPackage` | 研发协同域、管理信息化域 | 承接需求，拆解任务，约束资源和计划 | 是协同研发与管理信息化共享的桥对象 |
| `Task / Workflow / Review / Change` | 研发协同域、平台域 | 连接项目与执行、审批与版本 | 是流程引擎和任务引擎的核心对象 |
| `DataSet / Deliverable / File / Model` | 研发协同域、试验数据域 | 连接任务、产品、试验、知识 | 是数据谱系和数据治理的核心对象 |
| `Tool / APP / Template` | 平台域、研发协同域、试验数据域 | 被任务调用，处理数据，产出结果 | 是工业 APP 和工具流平台化的载体 |
| `Test / Measurement / Channel / Device / Site` | 试验数据域 | 形成采集链、分析链和站点网络 | 是 TDM 与 OT 集成的核心对象 |
| `KnowledgeAsset` | 知识与智能域 | 关联需求、任务、模型、报告、标准 | 是 AI 能力落地的基础资产 |
| `Contract / Equipment / Payment / Borrow / Scrap` | 管理信息化域 | 与项目、预算、组织、审批关联 | 体现管理域不是外围系统，而是业务闭环组成部分 |
| `Organization / User / Role / Dictionary / MasterData` | 平台域 | 横切所有域 | 必须做成共享内核而不是各域重复维护 |

## 4.3 建议采用的限界上下文

### A. 共享内核

- 组织、人员、岗位、角色
- 字典、编码、密级、权限
- 文件、对象存储、版本、基线
- 统一通知、消息、搜索
- 连接器与外部系统注册

### B. 研发协同上下文

- 需求管理
- 项目/WBS/工作包管理
- 任务执行与工作中心
- 设计/仿真/试验过程协同
- 研发门户与看板

### C. 试验数据上下文

- 试验建模
- 试验采集控制
- 数据中心/数据仓
- 分析与可视化
- 跨站点与跨网络数据链路

### D. 管理信息化上下文

- 科研计划
- 科研项目
- 科研合同
- 设备器材
- 主数据同步与调度

### E. 知识智能上下文

- 知识库
- 标准库
- 图谱
- 语义检索
- 智能推荐与大模型应用

## 4.4 端到端主业务链

### 链路 1：需求牵引的研发链

`需求/指标 -> 项目 -> WBS/工作包 -> 任务 -> 设计/仿真/试验 -> 数据中心 -> 评审/变更 -> 知识沉淀`

这是 `CDM` 最清楚的主链路，也是后续底座设计应该默认优先支持的“黄金路径”。[E1][E2][E16]

### 链路 2：数据驱动的试验链

`试验站点/设备 -> 原始采集 -> 文件/时序数据 -> 建模标准化 -> 数据中心 -> 分析/可视化 -> 虚实融合 -> 协同研发消费`

这是 `TDM` 最清楚的主链路，也是决定存储架构、接入架构、边缘架构的关键输入。[E4][E17][E18]

### 链路 3：管理驱动的经营链

`年度计划 -> 科研项目 -> 合同/经费 -> 设备资源 -> 审批与统计 -> 管理看板`

这条链路确保“管理信息化”不是与研发脱节的独立岛，而是同一底座上的管理域。最适合与 `项目/WBS/组织/主数据` 做深度融合。[E5][E6][E7][E8]

### 5.1 竞品带来的三点共识

1. **对象模型能力已成为底座分水岭**  
   华为、易立德、杰为都不再满足于“流程平台”，而是强调模型驱动、语义网络、XBOM、图谱、主线。

2. **流程能力仍是工业研发平台的刚需，但已不能单独构成壁垒**  
   索为和炎黄盈动代表了流程型平台的两种路径：工业研发深度版与通用低代码版。前者强业务，后者强效率。

3. **试验/仿真是我方真正的差异化突破口**  
   安世擅长 SPDM，但更偏仿真；我方如果把 `TDM + OLink + CDM` 打通，有机会形成“设计-仿真-试验-管理”一体化底座。

## 6. 对“底座架构设计与治理”的直接输入

## 6.1 建议的底座目标定位

建议将底座定义为：

> 面向工业研发场景的统一对象模型与统一协同执行平台，向上同时支撑协同研发、数字化试验、管理信息化，并通过数据、模型、知识和集成能力沉淀为企业级工业研发底座。

这一定义比“低代码平台”更贴近现有资产，也比“PLM 替代”更贴近你要做的事。

## 6.2 建议优先固化的底座能力域

### 1. 统一对象与元模型域

- 统一对象标识
- 统一元模型/扩展模型
- 生命周期、版本、基线、状态
- 模型发布、模型继承、模型扩展

### 2. 协同执行域

- 工作流/任务流/数据流/工具流
- 工作包/WBS/任务模板
- 评审、变更、通知、待办
- 跨角色、跨部门、跨专业协同

### 3. 数据治理域

- 文件、文档、对象、图谱、时序/测量数据统一纳管
- 数据谱系、成套性、影响分析、比对
- 搜索、标签、权限、密级、审计

### 4. 试验数据与边缘接入域

- OT 协议接入
- 边缘缓存/断点续传
- 多站点、多试验区汇聚
- 原始数据解析、标准化和回写

### 5. 集成域

- API 网关
- 事件总线
- 文件交换
- IT 系统连接器
- OT/工业协议连接器
- 外部工具/脚本/APP 接入

### 6. 知识智能域

- 知识库与标准库
- 图谱服务
- 语义检索
- 推荐/一致性检查
- 大模型辅助，但不侵入主交易链路

### 7. 运行治理域

- 部署拓扑标准
- 可观测性
- 服务治理
- 多环境配置
- 安全与国产化适配

## 6.3 建议的存储治理矩阵

| 数据类型 | 建议主存储 | 典型业务 |
|---|---|---|
| 主数据、项目、合同、审批、权限 | 关系型数据库 | 管理信息化、主数据、平台元数据 |
| 设计/试验业务对象扩展属性 | 文档数据库/宽表 | 协同研发、TDM 非固定模式对象 |
| 谱系、关系、主线、知识图谱 | 图数据库 | 数据追溯、影响分析、知识图谱 |
| 原始文件、模型、报告、多媒体 | 对象存储/文件存储 | 图档、报告、采集文件、附件 |
| 时序/测量数据 | 时序库/列式库 | 采集曲线、传感器数据、监测数据 |
| 搜索索引 | 搜索引擎 | 全文检索、标签搜索、数据黄页 |

## 6.4 架构治理优先级

### P0：先统一“词”和“对象”

- 统一产品命名体系
- 统一业务对象清单
- 统一 ID、版本、状态、编码规则
- 统一主数据口径

### P1：再统一“能力”

- 主数据服务
- 文件/对象服务
- 流程与任务服务
- 集成服务
- 搜索服务
- 权限与密级服务

### P2：最后统一“界面和产品包装”

- 统一门户
- 统一菜单与导航
- 统一看板
- 统一工作台交互风格

## 6.5 我建议你下一步直接产出的 6 个架构成果

1. **统一能力地图**  
   把 `CDM / TDM / 管理信息化` 三条线按能力域拆开，找出共享能力和专属能力。

2. **统一对象清单**  
   至少梳理：需求、项目、工作包、任务、数据集、模型、知识、试验、设备、站点、合同、组织、角色、权限。

3. **统一集成清单**  
   把 `PDM/PLM/TC/TDM/门户/主数据/需求系统/HPC/CAD/CAE/MATLAB/OT 协议` 统一登记为连接器目录。

4. **统一存储矩阵**  
   明确关系库、图库、时序库、对象存储、搜索引擎的职责边界。

5. **统一治理规则**  
   先从版本、状态、基线、谱系、密级、权限、归档开始。

6. **统一部署基线**  
   先定义标准化部署拓扑、可观测性基线、国产化兼容边界和运维责任矩阵。
