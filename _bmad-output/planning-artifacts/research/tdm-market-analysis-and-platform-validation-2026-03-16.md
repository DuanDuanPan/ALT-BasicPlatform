## 二、市场对平台/产品提出的八大核心要求

> **核心判断：市场不再需要一个"试验数据管理工具"，市场需要的是一个"工业试验数字化平台"。**

### 2.1 全域数据管理能力

> **试验数据从产生到归档销毁，全类型、全流程、全生命周期的数据，都要管得住。**

- 出现频率：14/16
- 客户的试验数据包括实时时序曲线、手工表单、视频/图片、仿真文件、Word/Excel报告等，格式不同、来源不同、密级不同、生命周期不同，要求一个系统统一管理
- 对平台的要求：必须支持多种存储引擎（关系型、时序、文档、对象、列式……）；必须有统一的数据存储适配层；必须支持数据全生命周期管理和存储分层

### 2.2 多源异构设备接入

> **不管什么品牌、什么协议的试验设备，平台都要能接进来、采到数。**

- 出现频率：12/16
- 反复出现的协议清单：CAN、OPC UA、Modbus、MQTT、EtherCAT、PROFINET、PXI、GPIB、RS232/485等
- 对平台的要求：必须有开放的协议适配框架，支持插件式接入新协议；必须支持高频采集（通道数≥2000，频率≥100Hz）；必须支持边缘/离线采集

### 2.3 工控网与办公网安全隔离互通

> **试验现场和办公区物理隔离，数据要能安全地"摆渡"过去。**

- 出现频率：6/16
- 军工试验场典型架构为工控网与办公网物理隔离，通过网闸、光盘摆渡或单向传输设备连通
- 对平台的要求：必须有产品级的两网数据同步架构；必须支持单向/双向网闸、串口同步、光盘三合一交互等多种摆渡方式；必须有审计日志和一致性校验

### 2.4 试验业务全流程管理

> **从试验计划、任务分派、资源调度到执行监控、报告生成，整条链路都要管起来。**

- 出现频率：13/16
- 客户要的不只是"管数据"，更是"管业务"——项目立项、任务分解、人员设备排班、执行监控、结果审批归档
- 对平台的要求：必须有可配置的流程引擎（BPMN2.0）；必须支持任务多级分解和资源管理；必须支持报告模板化+数据自动填充；流程模板可复用

### 2.5 数据分析与智能化

> **不仅要存数据，更要从数据中挖出结论——从基础统计到AI智能分析。**

- 出现频率：11/16（基础分析）、3/16（AI智能化）
- 客户对分析能力的三个层次要求：
  - 基础工程分析（FFT、滤波、统计、拟合等）——标配
  - 交互式探索分析（自定义流程、多组对比）——需灵活性
  - AI智能分析（知识图谱、智能问答、异常检测、报告自动生成）——趋势
- 对平台的要求：基础算法库内置开箱即用；必须有扩展机制（Python脚本等）；AI能力可选接入，平台不能封死这条路

### 2.6 数据追溯与谱系管理

> **任何一个试验结论，都要能追溯到原始数据、试验条件、操作人员和设备状态。**

- 出现频率：10/16
- 军工客户的刚性需求——问责链条必须完整：报告→分析结果→原始数据→采集设备→试验方案→需求来源
- 对平台的要求：必须有系统化追溯机制，不能靠人工记录；追溯关系要跨对象类型；追溯信息要自动生成；最好可视化呈现

### 2.7 安全合规与信创适配

> **涉密环境下运行，国产软硬件上部署，三员管理+审计日志一样不能少。**

- 出现频率：13/16（涉密资质）、10/16（国产化适配）
- 对平台的要求：三员管理（系统/安保/审计员），三权分立；完整操作审计日志；支持国产OS、数据库、中间件；数据分级保护；AI须支持国产芯片（昇腾/海光）

### 2.8 可扩展性与行业适配

> **平台要能快速适应不同行业、不同客户的个性化需求，而不是每个项目都从头定制。**

- 出现频率：普遍要求
- 16份文件来自六大行业，每个行业的试验类型、数据模型、流程不同，但底层需求高度相似
- 对平台的要求：行业模型预置能力；灵活扩展机制（扩展字段、增加算法、调整流程）；行业模型升级不影响客户扩展；标准化的打包/部署/升级机制

## 三、八大要求对平台/产品的分工映射

### 3.1 TDM产品 vs 底座 vs AI平台的责任划分

| 市场要求 | 满足方 | 为什么必须是它 |
|---------|--------|--------------|
| ① 全域数据管理 | TDM数据管理 + **底座多存储引擎** | 9种存储的统一适配需要底座级的抽象层 |
| ② 多源设备接入 | TDM采集测控层 | 15+协议已在TDM实现，底座提供Edge框架增强 |
| ③ 两网安全隔离 | TDM两网融合模块 | 已有产品级实现 |
| ④ 业务全流程管理 | TDM业务管理 + **底座流程引擎** | 可配置BPMN引擎是底座级共性能力 |
| ⑤ 分析与智能化 | TDM分析模块 + **底座Extension** + **AI平台** | 三层协作：内置算法+可扩展框架+AI增强 |
| ⑥ 数据追溯与谱系 | **底座TraceLink + Graph Service** | 跨对象、跨系统的图谱级追溯只有底座能做 |
| ⑦ 安全合规与信创 | **底座统一安全体系** + AI平台信创适配 | ACL/三员/审计/分保是跨产品共性需求 |
| ⑧ 可扩展与行业适配 | **底座ITP + Extension Unit + 柔性建模** | 模板打包/升级/扩展是平台级机制 |

**8项要求中，TDM产品独立可满足3项（②③和部分①④），其余5项必须底座参与才能完整交付。**

| 市场要求 | 没有底座 | 有了底座 |
|---------|---------|---------|

| ④ 业务全流程 | 每客户开发流程逻辑 | 底座BPMN引擎，流程可配置 |
| ⑤ 分析与智能化 | 内置算法，扩展靠开发 | Extension开放扩展+AI灵活接入 |
| ⑥ 数据追溯 | 靠命名规范+人工维护 | **TraceLink自动构建追溯图谱** |

## 八、底座对TDM的赋能价值

### 8.1 底座升级的战略定位

底座不是"给TDM补功能"，而是**让TDM从工具升级为平台**：

- 存储层 → 下沉到底座，TDM通过适配器调用
- 安全层 → 下沉到底座，CDM/TDM/MIS共享
- 追溯 → 下沉到底座Trace Service，TDM发出TraceIntent
- 分析算法 → 注册为ServerUnit，底座管理生命周期
- 行业模型 → 打包为ITP，底座管理安装/升级
- **TDM聚焦**：采集测控、业务管理、数据分析、指挥监控等核心试验业务

### 8.2 四层架构能力矩阵

| 层级 | 颜色 | 能力内容 | 属性 |
|------|------|---------|------|
| **TDM产品层** | 蓝色 | 15+协议、9存储、9算法、9行业、两网融合、指挥监控 | 已有成熟产品 |
| **底座赋能层** | 金色 | TraceLink追溯、ITP模板包、Extension Unit、柔性建模、流程引擎、统一安全 | 底座新增能力 |
| **AI智能化层** | 紫色 | 知识图谱、RAG问答、智能文档、多Agent协作、昇腾信创适配 | 已有独立产品，灵活集成 |
| **跨域贯通层** | 红色 | CAE仿真 ←→ TDM试验 ←→ CDM协同研发 | 全国唯一，不可复制 |

### 8.3 底座解锁的增量价值

| 增量价值 | 说明 |
|---------|------|
| 可投标范围扩大 | 覆盖率从65%提升至100%，原来"做不了"的项目变成"可以投" |
| 交付成本下降 | ITP行业模板包让9大行业模型从"项目定制"变"产品交付" |
| 利润率提升 | 底座统一安全/审计/存储，不再每个项目重复建设 |
| 竞争维度升级 | 从"TDM功能PK"升级为"平台生态PK"，瑞风无法跟进 |
| 大标竞争力 | TraceLink + CDM贯通，可投"需要多系统集成"的大型项目 |
| 客户粘性增强 | Extension开放扩展降低客户"被锁定"的顾虑 |
