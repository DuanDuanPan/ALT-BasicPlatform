## 第 2 页：为什么现在不定，后面还会回到项目制拼装

- **标题：** 现在最缺的不是功能，而是统一的资产交付机制
- **视觉建议：** 左侧"旧模式"（项目制拼装流程），右侧"后果"（红色标注），中间断裂箭头
- **内容：**
  - **现状：** 我们已经有 CDM / TDM / MIS 三条产品线，也有建模工具、流程配置、数据管理等基础能力
  - **真正的问题：**
    - 这些能力输出的是零散配置，不是可安装、可治理的业务资产
    - 每个项目仍然是公司侧按项目单独建模开发，现场再导入、调整和验证
    - 领域专有功能沉在项目方案里，难以沉淀为标准产品能力
  - **直接后果：**
    - 换一个客户，核心工作大多重来
    - 平台升级容易牵动客户个性化配置
    - 进入相邻行业场景时，仍需从单项目方案重新改造
  - **一句话结论：** 如果不先把资产交付机制定下来，后续每做一个项目都还会回到拼装模式
- **建议口播：**
  > 过去几年我们在产品能力上做了大量积累，但交付方式没有根本改变。每个项目看似在复用，实质上仍然是为单个客户生产一次性方案。问题不在工具不够，而在于缺少把这些工具产出统一收敛为可治理资产的机制。

## 第 3 页：这次到底定了什么，不定什么

- **标题：** 蓝图核心主张：一个平台内核 → 标准领域包 → 客户受控差量
- **视觉建议：** 一张简化总图（三层：平台内核 / 领域标准包 / 客户差量），不展开技术细节
- **内容：**

  **这次定的（已形成架构共识）：**
  - 平台内核统一共享能力（身份、权限、对象模型、文件、流程、任务、搜索、审计、集成）
  - 业务资产以 ITP（行业模板包）为载体：包含对象模型、活动模板、视图、运行时要求和合规规则
  - 标准化与差异化通过 M 轴分层共存：M0 平台内核 → M1 领域标准包 → M2 客户差量
  - 客户差量是在标准包之上叠加，不是直接改坏标准包
  - 运行时谁管什么状态明确分工：领域聚合管业务真相、编排层只管编排实例、审批只管审批

  **这次不定的（留给后续验证）：**
  - 具体技术选型和中间件
  - 详细容量数字和合规逐条映射
  - 扩展运行时的脚本引擎选型（需 POC 验证）
  - 编辑态工作台的 UX 细节

- **建议口播：**
  > 这次蓝图不是一张面面俱到的技术全图，而是先把三件最关键的事定住：共享能力放在哪、标准资产怎么交付、客户差异怎么受控。具体的技术选型和实现细节，留到试点阶段在真实场景里验证。

## 第 4 页：一张总图 + 一个 TDM 场景

- **标题：** 以风洞试验为例：标准包导入 + 受控差量 + 自动主线
- **视觉建议：** 上半区简化总图（平台内核 → ITP → 客户差量），下半区风洞试验工作台中保真示意
- **内容：**

  **上半区——交付模型总图：**
  ```text
  ┌─────────────────────────────────────────────┐
  │  平台内核（共享能力）                          │
  │  身份 / 权限 / 对象模型 / 追溯 / 审计 / ...   │
  ├─────────────────────────────────────────────┤
  │  领域标准包（ITP）                             │
  │  风洞试验 ITP: 对象12个 / 模板8个 / 视图6个    │
  ├─────────────────────────────────────────────┤
  │  客户差量（M2）                                │
  │  A所: 涉密等级 / 特批编码 / 审批附加规则        │
  └─────────────────────────────────────────────┘
  ```

  **下半区——导入后的运行形态：**
  - 导入 ITP 后不是空平台，而是直接形成业务工作台
  - 任务、活动模板、审批、工具执行、数据采集都已经成型
  - 从需求到报告的追溯主线自动形成
  - 客户特有字段和规则以差量方式叠加，不改坏标准包

  **编辑态简示（一段口播即可，不展开整页）：**
  - 实施人员在运行界面上直接切换"设计模式"，即可在预设的扩展点上做差异化配置
  - 所有修改暂存后经影响分析再发布，保障生产环境安全

- **建议口播：**
  > 上面这张图就是整个交付模型的缩影。平台提供共享能力地板，领域团队把风洞试验的全套业务机制打成一个标准 ITP 包，客户的个性化需求以差量叠加上去。最终形成的"方案包"就是给这个客户量身装配的运行配置——哪个基线、装了哪些 ITP、做了哪些定制、连了哪些外部系统，一目了然。导入以后，实施现场直接得到一个可运行的业务工作台，不再从零拼装。如果客户需要进一步调整，实施人员可以在运行界面上直接进入设计模式，在预设的位置做受控扩展，所有修改先暂存、先检查影响再发布。下一个类似客户，可以直接克隆这个方案包再微调，不用从头装配。

## 第 5 页：为什么先选 TDM，怎么双线推进

- **标题：** TDM 风洞试验先行，底座修路与标杆域造车双线并进
- **视觉建议：** 左侧三域比选（定性理由，不用评分表），右侧双线路线图
- **内容：**

  **为什么建议 TDM 先行（三条理由）：**
  1. TDM 的对象模型最丰富，最能检验 ITP 包结构和模型注册能力
  2. TDM 天然具备从需求到报告的完整数据主线，可一次性验证追溯和影响分析
  3. TDM 对编排和外部集成的依赖最轻，不会因底座尚未收敛的编排能力而卡住试点

  **CDM / MIS 为什么不宜首选：**
  - CDM 编排和连接器依赖最重，底座侧编排能力尚在细化
  - MIS 规则极度客户化，差量机制尚未闭环验证

  **双线推进：**

  | 线 | 职责 | Phase 0 交付 |
  |---|------|-------------|
  | 底座线 | 把路修出来 | 对象模型注册、版本绑定、追溯基础能力、编辑态最小 POC |
  | 标杆域线 | 把第一辆车造出来 | 风洞试验标准 ITP + 装载验证 + 差量叠加验证 + 能力模块沉淀 |

  **后续扩展路径：** TDM 验证闭环后 → CDM 扩展编排和连接器能力 → MIS 扩展规则和合规能力

- **建议口播：**
  > TDM 是"模型最重、编排最轻、主线最完整"的域，最适合作为第一个标杆。底座线负责把共享能力的路修出来，标杆域线负责造第一辆车并把关键能力沉淀为产品资产。两条线相互验证，既避免闭门造车，也避免继续停留在项目制拼装。

## 第 6 页：成功标准、未决项、需要领导支持的事

- **标题：** Phase 0 成功标准与需要的支持
- **视觉建议：** 三栏——左"成功标准"、中"未决项"、右"需要的支持"
- **内容：**

  **Phase 0 成功标准：**
  - 工业语义主线最小闭环打通（需求→架构→试验→数据→追溯）
  - 复制交付最小闭环打通（ITP 导入→校验→差量→运行）
  - 首个领域标准包可重复装载到不同租户
  - 至少从标杆域项目中沉淀出可复用的 ITP 能力模块

  **主要未决项（需 Phase 0 验证后收敛）：**
  - 扩展脚本引擎的具体技术选型（多个候选方案待实测后收敛，详见专家附录）
  - 编辑态工作台的交互体验细化
  - 活动模板的标准规范
  - 版本兼容性检查和迁移机制

  **需要领导支持的事：**
  1. 认可底座作为统一方向，不再按单产品线分散推进
  2. 给出跨产品线协同资源，而不是只当某个团队的技术预研
  3. 同意建立平台级治理机制（模型治理、复制治理、安全治理）

- **建议口播：**
  > Phase 0 的成功不是看页面做了多少，而是看两条闭环能不能跑通、标准包能不能重复装载、领域能力能不能从项目里沉淀出来。还有几个技术选型需要在试点阶段通过实测来收敛，这些不影响方向拍板。最后需要领导支持的核心是——把这件事定性为公司级统一方向，给出跨产品线的协同资源。如果只当作某个团队的内部预研，底座的价值就发挥不出来。

## 领导大概率会追问的 5 个问题

### 问题 1：为什么不能直接在现有产品上继续做？

> 可以继续做，但如果不先统一蓝图，各产品线还是会各自演进，共享能力无法沉淀，复制交付成本下不来。蓝图不是替代现有产品，而是让现有产品逐步长到同一个底座上。

### 问题 2：为什么不直接上一个通用低代码平台？

> 通用低代码能解决表单和流程效率问题，但解决不了工业研发场景里的对象模型、数字主线、标准语义、边缘接入和复制交付问题。低代码只能放在受控装配层，不能替代底座。

### 问题 3：Phase 0 成功的标志是什么？

> 两件事跑通：工业语义主线闭环 + 复制交付闭环。标准包能重复装载，领域能力能从项目里沉淀出来。

### 问题 4：最大的风险是什么？

> 不是技术实现，而是跨产品线不按统一治理推进。如果没有统一对象模型、标准语义和复制治理，后面还是会碎片化。

### 问题 5：最直接的收益是什么？

> 短期：减少重复建设，提高项目交付一致性。中期：把现有资产沉淀为平台能力。长期：形成可复制、可扩展的工业研发底座产品。
