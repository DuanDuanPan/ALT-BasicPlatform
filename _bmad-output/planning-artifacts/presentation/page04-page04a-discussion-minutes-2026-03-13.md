# Page 04 / Page 04A 讨论纪要

- 日期: 2026-03-13
- 讨论方式: Party Mode 多角色收敛
- 参与角色: BMad Master, Architect, PM, UX Designer, Developer Agent
- 相关原型: `presentation.pen`
- 讨论主题: `Page 04A`、`Page 04`、`模块`、`能力包`、`页面`、`业务对象` 的关系与页面职责划分

## 一、讨论背景

围绕当前原型，讨论聚焦在以下问题：

1. `Page 04A` 是否应继续承担“基线继承决策”主叙事，还是转为索引入口。
2. `Page 04` 是否与新的对象关系模型存在职责冲突。
3. 当实施调整经常涉及一个模块下的多个页面时，平台如何支持“快速定位 + 影响分析 + 安全修改”。
4. `模块`、`能力包`、`页面`、`业务对象`、`变体`、`发布包` 的关系如何收敛。

## 二、已达成共识

### 1. 归属关系

- `业务对象` 的定义归属默认落在 `模块`。
- `页面` 的定义归属默认落在 `模块`。
- `能力包` 不直接拥有页面和对象。
- `能力包` 的职责是装配、暴露、交付模块资产，而不是成为页面/对象的定义者。

换句话说，应区分：

- `owning`：谁定义该资产
- `including`：谁把该资产装配带出

因此，正确表达应为：

- 页面定义在模块中，能力包通过装配关系暴露页面。
- 对象定义在模块中，能力包通过装配关系带出对象。

### 2. 特殊情形处理

如果存在跨多个模块的总览页、汇总页、装配页或包级协调对象，不建议直接将其挂到 `能力包` 下。当前建议引入一个单独概念来承载这类资产：

- `包级编排模块`

该概念在本轮讨论中被提出并被认为合理，但尚未最终固化为正式对象模型的一部分。

### 3. 主线定位方式

页面定位不应默认从“模块树”开始，而应优先从业务主线进入：

`业务主线 -> 能力包 -> 模块 -> 页面组/页面`

需要支持以下几种反查入口：

- 按业务主线找
- 按能力包找
- 按模块找
- 按业务对象找
- 按业务动作找

### 4. Change Scope 概念

实施时调整的对象不应被假定为“单个页面”。本轮收敛出一个中间概念：

- `Change Scope`

建议至少支持以下范围类型：

- `page`
- `page_set`
- `module`

`capability_pack` 暂时更适合用于查看和装配，不建议直接作为页面编辑的主对象。

## 三、对象关系收敛版

以下为当前讨论形成的收敛版对象关系：

```text
产品基线 ProductBaseline
  -> 预置/推荐能力包 CapabilityPack
      -> 由多个模块 CapabilityModule 组成
          -> 页面 PageTemplate
          -> 动作 ActionModel
          -> 业务对象 DomainObject
          -> 流程 FlowModel
          -> 规则/约束 RuleSet
          -> 连接器引用 ConnectorGroup

项目/单位级派生 ProductVariant
  -> 继承 ProductBaseline
  -> 应用 ITP
  -> 选择 CapabilityPack
  -> 绑定 ConnectorGroup
  -> 叠加 Delta
  -> 生成 ReleasePackage
```

补充口径：

- `模块` 是内部构件层。
- `能力包` 是装配/交付层。
- `页面` 和 `对象` 属于构件层，不属于装配层。

## 四、Page 04A 新定位

### 1. 页面定位

`Page 04A = 模块/页面扩展索引与影响分析页`

这意味着它不再是单纯的“页面索引页”，而是：

- 扩展定位入口
- 范围选择入口
- 影响分析入口
- 编辑跳转入口

### 2. 核心职责

`Page 04A` 负责：

- `Locate`：通过主线/能力包/模块快速定位
- `Scope`：判定改单页、改页面组还是改模块
- `Impact`：展示对象/动作/流程/规则/能力包/变体影响
- `Jump`：跳转到真正的编辑工作面

### 3. 建议页面结构

- 顶部: 上下文条
  - 当前基线
  - 当前变体
  - 当前能力包
  - 当前模块
- 左栏: 主线与能力入口
  - 业务主线
  - 能力包
  - 最近修改模块
  - 收藏模块
- 中间: 扩展定位台
  - 搜索
  - 筛选
  - 模块卡片
  - 页面组清单
  - Change Scope 选择
- 右栏: 影响分析
  - 受影响页面
  - 受影响对象
  - 受影响动作
  - 受影响流程/规则
  - 被哪些能力包暴露
  - 被哪些变体覆盖

### 4. 页面产出

`Page 04A` 的直接产出应为：

- `Change Scope`
- `Impact Report`
- `Jump Target`

它不应直接承担页面内容编辑。

## 五、Page 04 新定位

### 1. 页面定位

`Page 04 = 模块页面与动作建模页`

它是模块上下文内的编辑工作面，而不是平台级混合总控台。

### 2. 核心职责

`Page 04` 负责：

- 编辑页面模板
- 编辑页面组
- 编辑动作与联动
- 绑定对象、流程、规则
- 查看当前修改的模块级影响

### 3. 不再作为主职责承载的内容

以下内容不应继续作为 `Page 04` 的主叙事：

- 平台基线继承决策
- 运行变体总体决策
- 发布校验主流程
- 能力包归属编辑

这些内容要么前移到 `Page 04A` 的上下文/影响分析，要么后移到装配与预演阶段。

### 4. 建议页面结构

- 顶部: 面包屑
  - `基线 > 能力包 > 模块 > 页面组/页面`
- 左栏: 模块资产树
  - 页面组
  - 页面
  - 动作集
  - 共享组件
  - 模块级规则引用
- 中间: 编辑工作区
  - 模块共享视图
  - 页面组视图
  - 单页视图
- 右栏: 引用与影响
  - 被哪些能力包暴露
  - 被哪些变体覆盖
  - 依赖哪些对象/流程/规则
  - 风险提示

### 5. 必须支持的编辑粒度

为了满足实施场景，`Page 04` 不应只是单页编辑器，至少需要支持：

- `单页`
- `页面组`
- `模块共享`

### 6. 建议新增能力

- 批量调整多个页面的动作显隐
- 批量调整页面组共享区域
- 批量替换对象绑定
- 批量查看本次修改的影响范围

## 六、Page 04A 与 Page 04 的明确分工

推荐的 handoff 关系为：

```text
Page 04A
  输入: 主线 / 能力包 / 模块 / 搜索词
  输出: Change Scope + Impact Report + Jump Target

Page 04
  输入: Change Scope
  输出: 模块级页面与动作变更
```

一句话总结：

- `Page 04A` 负责找到该改哪里
- `Page 04` 负责真正去改

## 七、对当前原型的直接调整建议

### 1. Page 04A

当前版本为“基线继承决策页”，建议调整为“扩展索引与影响分析页”。

建议替换：

- 页面名:
  - 从 `基线继承决策页`
  - 改为 `模块/页面扩展索引与影响分析页`
- 副标题:
  - 从“优先从已发布基线派生方案；模块选择用于补充能力或构建新基线”
  - 改为“按业务主线、能力包、模块快速定位扩展范围，并预览对象/流程/规则影响”
- 中间主区:
  - 从“继承决策结论”
  - 改为“搜索 + 模块定位 + 页面组选择 + Change Scope”
- 右栏:
  - 从“结论建议/治理约束/待讨论问题”
  - 改为“影响分析/依赖资产/建议跳转”

### 2. Page 04

建议替换：

- 页面名:
  - 从 `视图与动作建模页`
  - 改为 `模块页面与动作建模页`
- 左栏:
  - 从 `平台能力树`
  - 改为 `模块资产树`
- 副标题:
  - 从 `平台基线 / 运行变体 / 字段矩阵 / 发布校验`
  - 改为 `页面模板 / 动作联动 / 对象绑定 / 模块影响`
- 页面语义:
  - 从“平台级混合工作台”
  - 改为“模块级资产编辑器”

## 八、暂未最终定项

以下问题已被识别，但尚未最终敲定：

1. `包级编排模块` 是否成为正式对象模型中的标准概念
2. `能力包` 是否允许存在仅装配级暴露、不具备独立编辑权的包级页面入口
3. `Page 04A` 是否保留少量“基线继承上下文”说明，还是完全转成索引型页面
4. `Change Scope` 后续是否需要扩展到 `capability_pack`

## 九、下一步建议

建议按以下顺序推进：

1. 先把 `Page 04A` 调整为索引与影响分析页
2. 再把 `Page 04` 调整为模块页面与动作建模页
3. 然后补一版严格归属的对象关系图
4. 最后再决定 `包级编排模块` 是否正式入模

## 十、结论摘要

本轮讨论的核心结论是：

- 页面和业务对象默认属于模块，不属于能力包
- 能力包负责装配和暴露，不负责定义
- 实施调整常常面向页面组甚至整个模块，而非单页
- 因此 `Page 04A` 应承担“定位 + 范围 + 影响 + 跳转”
- `Page 04` 应承担“模块上下文内的页面与动作编辑”

这两页需要从“概念混合”调整为“职责解耦”。
