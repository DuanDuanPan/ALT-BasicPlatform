# 模型化汇报材料更新方案

日期: 2026-03-16（基于 03-15 版本重构，吸收 03-16 概念精简共识）
用途: 设计如何在汇报 PPT 和原型中体现模型化架构的完整叙事

## 一、叙事主线

### 核心叙事（一句话）

> **从"工具交付"走向"资产交付"——构建一次，复制多次，客户安全定制，定制反哺产品。**

### 三层交付模型（精简后的概念骨架）

```text
方案包 (SolutionInstance)  ← 项目级：给某个客户装配的运行配置
  ├── 基线选择 (ProductBaseline P0)
  ├── 安装的 ITP ×N          ← 产品级：可复用的业务能力包
  │     └── 模块 ×N          ← 构建级：业务能力定义单元
  │           └── 资产        ← PageTemplate / ActionModel / OrchestrationModel / RuleSet / FED
  ├── M2 差量 (FED)           ← 客户定制
  └── 连接器绑定              ← 外部系统对接
```

**三个关键动词：**
- **构建** → 领域团队定义模块和资产，打包为 ITP
- **交付** → 安装 ITP + 绑定连接器 → 形成方案包
- **积累** → 客户定制（FED）→ 晋升 M1 → 更新 ITP → 飞轮

### 叙事线对比

```text
旧叙事（03-11，单维）：
  痛点 → 模型化定义 → ITP → 导入后长什么样 → M 轴 → 运行时 → 治理 → 拍板

新叙事（03-16，三层主线）：
  痛点 → 模型化定义 → 三层交付模型（方案包 / ITP / 模块）
    → ITP 里装了什么
    → 方案包：装配后客户看到什么
    → 编辑态：客户如何安全定制
    → 双向流动：定制如何反哺产品
    → 运行时边界 → 治理闭环 → 拍板
```

核心变化：**从"ITP + M 轴"的单维叙事，升级为"方案包 → ITP → 模块"的三层交付叙事。**

## 三、关键 Slide 内容设计

### Slide 3.5：三层交付模型——方案包 / ITP / 模块

**页面目标：** 用一张图建立"三层交付"心智模型，替代原来的"P0-P5 六层堆叠"（对非技术受众太重）

**视觉建议：** 三层嵌套图（外→内：方案包 → ITP → 模块），右侧 M 轴色带

**页面核心内容：**

```text
┌─────────────────────────────────────────────────────────────────┐
│  方案包 (SolutionInstance)                                      │
│  航天院-军检验收方案 v1.3                                        │
│  ┌────────────────────────────────────┐  ┌──────────────────┐  │
│  │  ITP: 风洞试验标准包 v2.0          │  │  ITP: 数据归档包  │  │
│  │  ┌──────────┐ ┌──────────────────┐│  │  v1.1            │  │
│  │  │ 模块:     │ │ 模块:            ││  │  ┌────────────┐  │  │
│  │  │ 试验管理  │ │ 数据采集与处理   ││  │  │ 模块:       │  │  │
│  │  │ (12 对象  │ │ (8 对象          ││  │  │ 归档与卷宗  │  │  │
│  │  │  6 页面   │ │  4 页面          ││  │  │ (5 对象)    │  │  │
│  │  │  3 编排)  │ │  2 编排)         ││  │  └────────────┘  │  │
│  │  └──────────┘ └──────────────────┘│  └──────────────────┘  │
│  └────────────────────────────────────┘                         │
│  + M2 差量: 涉密等级 / 特批编码 / 3 条附加审批规则               │
│  + 连接器: 风洞数采系统 API / 试验仪器 SCADA                     │
└─────────────────────────────────────────────────────────────────┘
```

**右侧 M 轴标注：**
```text
M0  平台内核（引擎 + 组件）      ← 平台团队，年级更新
M0.5 公共业务对象                 ← 平台团队，半年级
M1  领域标准 ITP                  ← 领域团队，季度级
M2  客户差量 FED                  ← 客户/实施，月级
方案包 = M0 + M0.5 + M1 ITP + M2 FED + 连接器
```

**要讲透的 3 个点：**
1. 方案包是给客户的最终交付物——它不是一个 ITP，而是多个 ITP + 客户定制 + 连接器的组合
2. ITP 是可复用的标准能力单元——换一个客户，ITP 不变，只改方案包里的 M2 差量
3. 模块是 ITP 的内部构件——领域团队在这里定义业务对象、页面、编排和规则

**建议口播：**
> 这张图就是整个交付模型的"地图"。最外层是方案包——给航天院装配的完整运行配置。它里面装了两个标准 ITP：风洞试验包和数据归档包。每个 ITP 又由若干模块组成，模块里是具体的业务对象、页面模板和编排规则。方案包还包括客户的差异化定制和外部系统连接。下一个客户如果业务类似，可以直接克隆这个方案包再微调——ITP 不用重做，只改差量部分。如果多个客户都做了相似的定制，领域团队就可以把这些定制提炼为标准 ITP 能力——这就是飞轮效应的起点。

### Slide 6.5：双向流动——定制如何反哺产品

**页面目标：** 用方案包视角讲清飞轮

**视觉建议：** 循环流动图，突出两条复用路径

**页面核心内容：**

```text
         ┌──── 平台能力扩大 ←──── 跨领域共性提炼 ─────┐
         │                                              │
         ▼                                              │
   ┌──────────┐                                  ┌──────────┐
   │  平台     │ ── 引擎+组件+扩展点 ──→          │  产品     │
   │  M0/M0.5 │                                  │  M1 ITP  │
   └──────────┘                                  └──────────┘
                                                        │
                        安装 ITP ──→ 形成方案包          │
                                                        │
                              ┌──────────┐              │
                              │  方案包   │  ←── ITP ───┘
                              │  M2 实施  │
                              └──────────┘
                                    │
                          ┌─────────┴─────────┐
                          │                   │
                    快路径：clone         慢路径：promote
                    方案包 → 下一个客户   FED → M1 → ITP v2
```

**两条复用路径一句话：**
- **快路径（clone）：** 方案包整体克隆给下一个类似客户，调整差异即可上线
- **慢路径（promote）：** 多项目的相似 FED 提炼为标准 ITP，所有客户受益

**建议口播：**
> 飞轮有两条路径在转。快路径是直接克隆方案包——A 所做完了，B 所业务类似，直接复制方案包改差量就能上线。慢路径是把多个客户共同做过的定制提炼为标准 ITP 能力——这需要领域团队评审和沉淀，但一旦进入标准包，所有客户都受益。两条路径不冲突：先用快路径赶交付，再用慢路径建产品。项目做得越多，ITP 越完整，下一个项目交付越快。

- "方案包起始/复制"→ SolutionInstance.clone()
- "查看方案包结构"→ 展示 baselineRef + installedITPs + appliedFEDs + connectorBindings

### Page 08 与装配流程的对齐

Page 08 方案装配台的操作步骤现已明确映射到方案包属性：
```text
选择平台基线版本  → baselineRef

绑定连接器        → connectorBindings[]

预演              → status: staging
发布              → status: published
```
