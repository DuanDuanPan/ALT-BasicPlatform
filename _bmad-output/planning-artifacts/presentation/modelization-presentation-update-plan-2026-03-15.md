## 三、新增 Slide 内容设计

### Slide 3.5：完整层次结构——从平台到项目的六层架构

**页面目标：** 一张图让所有听众建立统一的架构心智模型

**视觉建议：** 六层横向堆叠图，从 P5（底）到 P0（顶），每层标注定位和关键对象

**页面核心内容：**

```text
┌──────────────────────────────────────────────────────────┐
│ P0  产品基线        OLTran.CDM / TDM / MIS              │
├──────────────────────────────────────────────────────────┤
│ P1  能力包          project-execution / evidence-dossier │
│                    + 包级编排模块 + FED 协调              │
├──────────────────────────────────────────────────────────┤
│ P2  模块            work-package-center / task-mgmt      │
├──────────────────────────────────────────────────────────┤
│ P3  资产            PageTemplate / DomainObject /        │
│                    ActionModel / FlowModel / FED         │
├──────────────────────────────────────────────────────────┤
│ P4  构件            ExtensionPoint / FieldBinding /      │
│                    DesignChangeset / TemplatePatch        │
├──────────────────────────────────────────────────────────┤
│ P5  引擎与组件      graph-canvas / form-engine /         │
│                    io-configurator / relation-browser     │
└──────────────────────────────────────────────────────────┘
```

右侧叠加 M 轴色带：M0 蓝 / M0.5 青 / M1 绿 / M2 橙

**要讲透的 3 个点：**
1. 每一层有明确的职责边界——定义者 vs 装配者 vs 运行者
2. ITP 不是新概念，它就是 P1-P3 的打包产物
3. M2 编辑态（设计模式）操作的是 P3-P4 层，由 P5 引擎兜底

**建议口播：**
> 这张图是理解整个底座架构的"地图"。从底往上看：P5 是平台的交互引擎，硬编码保证品质；P4 是资产的内部结构，编辑态在这里操作；P3 是模块的内容物——页面、对象、流程、规则和特性扩展；P2 是模块，业务能力的定义单元；P1 是能力包，面向客户的交付单元；P0 是产品基线，出厂形态。ITP 就是把 P1 到 P3 打成一个包。右侧的颜色代表谁控制——蓝色是平台团队，绿色是领域团队，橙色是客户和实施。

### Slide 5.5：编辑态演示——实施顾问如何原地扩展能力

**页面目标：** 用原型 Page 04 的编辑态截图，让听众直观看到"扩展一个字段"的完整过程

**视觉建议：** Page 04 原型截图 + 4 个标注气泡

**页面结构：**
- 中间：Page 04 编辑态截图（已有橙色虚线装饰器的版本）
- 4 个标注：
  1. 🔧 设计模式开关 → "一键进入编辑态"
  2. ⚙️ 虚线边框 + 齿轮 → "可配置区域自动高亮"
  3. 右栏属性面板 → "选中元素即可配置字段和动作"
  4. Draft 暂存条 → "变更暂存，影响分析后才发布"

**要讲透的 4 个点：**
1. 不需要独立的"建模工具"——运行界面就是配置界面
2. 引擎（图形画布、表格等）保证交互品质，Schema 保证灵活扩展
3. ExtensionPoint 控制"哪里可以改"——受控灵活，不是无限灵活
4. Draft → 影响分析 → 发布，保障生产环境安全

**建议口播：**
> 这个原型展示的是实施人员的日常操作。他不需要进入一个独立的"建模工具"，而是直接在业务工作台上点击"设计模式"。橙色虚线标记的区域就是可以扩展的地方——这些扩展点是由领域团队预埋的，客户只能在预埋的位置做扩展。选中一个表格，右栏立刻出现列配置。加完字段后，变更暂存在 Draft 里，经过影响分析确认无冲突后才发布到运行态。整个过程所见即所得。

### Slide 6.5：双向流动——项目积累如何反哺产品和平台

**页面目标：** 讲透"飞轮效应"——这是商业模式的核心

**视觉建议：** 循环流动图（平台 → 产品 → 项目 → 产品 → 平台）

**页面核心内容：**

```text
         ┌──── 平台能力扩大 ←───── 跨领域共性提炼 ─────┐
         │                                              │
         ▼                                              │
   ┌──────────┐                                  ┌──────────┐
   │  平台     │ ── 引擎+组件+扩展点 ──→          │  产品     │
   │  M0/M0.5 │                                  │  M1 ITP  │
   └──────────┘                                  └──────────┘
         │                                              │
         │ 支撑                                   交付  │
         │                                              │
         │         ┌──────────┐                         │
         └────→    │  项目     │  ←──── 能力包 ─────────┘
                   │  M2 实施  │
                   └──────────┘
                         │
                   编辑态 → FED
                         │
                   多项目 FED → M1 提炼标准
                   多领域模式 → M0 抽象通用
```

**一句话核心：** 每一个客户项目的 M2 定制，都是平台未来能力的"增量积累"。

**建议口播：**
> 这是整个模型化架构最重要的商业逻辑。从上往下看，平台提供引擎和扩展点，领域团队用它构建标准产品能力，客户项目在此基础上做差异化扩展。从下往上看，客户的每一次定制都自动生成结构化的特性扩展记录。当多个项目做了相似的扩展，领域团队就可以提炼为标准能力包。当多个领域出现相似模式，平台团队就可以抽象为通用引擎或组件。这是一个越转越快的飞轮——项目做得越多，产品越完整，下一个项目交付越快。
