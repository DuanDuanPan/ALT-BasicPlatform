# 数字化验收平台 12 页原型内容说明

状态：`Draft v0.1`  
日期：`2026-03-12`  
用途：沉淀“军检典型场景”原型分镜内容，作为后续 Pencil 高保真页面绘制的直接输入。

---

## 1. 原型目标

本原型不是单纯展示某个业务页面，而是要让用户直观看到：

**平台基线 + 验收方案 ITP + 能力模块 + 连接器组 + 客户差量，如何从建模一步步进入可运行应用。**

因此，原型应按“先建模、后装配、再发布、最后运行”的顺序展开。

---

## 2. 绑定场景

### 2.1 典型场景

选取军检典型业务主线：

**面向多型号、多批次产品的数字化军检验收全链路场景，从提交计划生成，到资料审查、抽样推送、实物检验、证据包装配、电子卷宗归档、追溯与态势分析。**

### 2.2 统一故事主线

为保证 12 页原型不是孤立页面，统一绑定以下示例上下文（仅用于原型演示，不代表产品范围限制）：

- 示例型号：`HJ-2403`
- 示例批次：`批次 A-17`
- 验收类型：`固定军检 + 产品交付验收`
- 主要角色：
  - 质量部
  - 客户代表
  - 计划部
  - 生产单位
  - 工艺研究所
  - 产品设计所
- 主要外部系统：
  - `PDM`
  - `MES / 生产综合`
  - `主数据管理`
  - `数据中台`
  - `TDM`
  - `音视频监控`
  - `档案系统`
  - `电子履历本`
  - `数字化检验检测`
  - `IIOT`

## 3. 平台口径

这套原型必须体现以下平台思路：

1. 先定义应用，再定义对象、页面、流程和连接器。
2. `ODS/TODS` 只承接试验语义层，不接管整个平台对象体系。
3. 页面、按钮、表单、流程、连接器都应首先表现为正式资产，而不是项目硬编码。
4. `ITP` 负责解决方案装配；`capability module` 负责能力复用；`M2 localDelta` 负责客户差量。
5. 权限、审计、密级、`TraceLink`、发布治理贯穿所有页面。

## 4. 12 页分镜

## Page 01 方案总览与启动页

### 页面目标

让用户一眼理解：当前打开的不是一个业务模块，而是一套军检验收方案应用。

### 页面定位

功能页，不是海报页。  
页面应允许：

- 选择军检方案
- 查看当前方案装配结果
- 进入建模工作台
- 查看运行应用入口

### 页面结构

- 顶部：方案名称、版本、批次、发布状态
- 左侧：方案步骤导航
- 中部：方案总装视图
- 右侧：当前装配结果与风险提示
- 底部：从建模到运行的链路状态

### 核心控件

- 方案选择下拉框
- 批次输入框
- 版本只读输入框
- 状态标签
- “进入应用建模”“进入运行应用”按钮
- 装配结果卡片
- 风险告警卡片

### 页面文案建议

- 标题：`军检验收方案总览`
- 副标题：`面向多型号、多批次的数字化验收方案装配与运行总控（当前示例：HJ-2403 / 批次 A-17）`

### 数据绑定

- 当前选择的方案 `ITP`
- 已启用模块列表
- 连接器健康状态
- 客户差量数量
- 发布状态

## Page 03A 试验字段语义映射交互页

### 页面目标

用一个具体试验字段示例，让用户理解“字段名”如何被映射成可交换、可分析、可追溯的标准语义资产。

### 页面定位

功能页。  
用户应能在这里：

- 选择待映射字段
- 查看字段别名与外部来源
- 拆解字段语义
- 选择标准标签
- 归一单位
- 预览映射影响
- 生成 `SemanticBinding`

### 示例字段

- 本地字段：`engineInletTemp`
- 中文名称：`发动机进口温度`
- 外部别名：`temp_in`、`CH_07`
- 目标语义：`Temperature`
- 测点位置：`Engine Inlet`
- 单位归一：`°C -> degC`

### 页面结构

- 左侧：待映射字段与步骤导航
- 中部：语义拆解工作区 + 映射结果预览
- 右侧：校验结果 + 影响分析 + 发布输出

### 核心控件

- 字段候选列表
- 别名选择项
- 语义类别卡
- 测点位置卡
- 单位归一卡
- 映射规则表
- 校验结果卡
- “生成映射”按钮

### 关键强调

- 业务字段名只解决“本地可读”，语义映射解决“跨系统可理解”
- `engineInletTemp`、`temp_in`、`CH_07` 可被归并为同一温度测量语义
- 映射结果应输出正式资产：`SemanticBinding / AliasMap / UnitNormalizationRule`

## Page 04 视图与动作建模页

### 页面目标

把门户、工作台、列表、表单、查询区、图表区、抽屉、弹窗，以及关键按钮动作，定义成正式资产。  
让同一领域对象可以按阶段、角色、任务类型维护多套 UI 模板，并通过继承与局部覆盖完成客户化装配。  
以“一级导航 = 批次质量工作台”为例，展示如何把 `Page 11 证据包与电子卷宗页` 和 `Page 12 追溯与态势分析页` 动态组合成一套复合工作台。  
同时展示如何在平台“证据包 / 卷宗 / 追溯 / 态势”领域能力之上，新增 3 个扩展字段并把它们落到列表、查询区、抽屉表单和图表里，而不是绕开平台能力另起一套页面。

### 页面定位

功能页。  
用户应能在这里：

- 选择绑定的领域对象
- 选择或新建 UI 模板
- 查看平台基线模板、领域能力包与客户 patch 的继承链
- 通过壳模板、容器模板、组件模板组合一级导航工作台
- 把左侧树、中间多 Tab、Tab 内列表、按钮弹抽屉表单、右侧图表组合成一套工作台
- 从对象字段面板拖拽字段到表格、表单、查询区、图表
- 配置字段的显示、必填、只读、校验、排序、筛选、聚合等 UI 行为
- 维护模板变体，例如按阶段、角色、任务类型拆分多套模板
- 配置模板继承、局部覆盖与差异提示
- 在平台领域功能前提下追加客户字段、客户动作与客户约束
- 选择动作模板并配置动作上下文映射
- 将字段引用到动作、规则、校验与后续联动中
- 明确扩展字段如何进入新增/修改抽屉、查询区、列表列和图表维度
- 查看模型变更后的提醒、更新建议与手动处理项

- 中部：以“批次质量工作台”为例的组合式页面编辑器画布
- 右侧：字段、动作、patch 与继承编辑器
- 底部：模板变体、继承关系与变更提示

### 页面原则

- 领域模型只定义业务语义，不直接携带 UI 布局信息。
- 扩展建模必须以前台领域能力包和平台基线模板为父模板，不鼓励从零另建同类页面。
- 新增字段默认隐藏，只有被模板显式勾选、拖入或绑定后才会出现在 UI 中。
- 一级导航允许由多个子页面或子视图片段动态组合，不要求“一导航只对应一页面”。
- Tab、列表、查询区、抽屉表单、图表都属于同一套工作台模板的可编排部件。
- 同一对象允许按阶段、角色、任务类型维护多套模板，运行时按条件切换。
- 模板覆盖采用“继承 + 局部 patch”，不鼓励整套复制分叉。
- 模型变更后不强制自动改模板，而是给出提醒、更新建议和手动处理入口。
- 动作上下文映射、规则引用与模板字段绑定保持一致，避免页面与动作脱节。

### 建议在页面中直接展示的扩展示例

- 一级导航：`批次质量工作台`
- 左侧树：资料分类树 / 追溯对象树
- 中间 Tab：
  - `证据清单`，来自 Page 11，包含列表和行按钮
  - `卷宗预览`，来自 Page 11，包含详情卡和归档状态
  - `追溯分析`，来自 Page 12，包含查询区、主线图谱和右侧 ECharts
- 行按钮：`发起替换审批`，点击后弹出抽屉表单
- 右侧图表：质量风险趋势、齐套率分布、归档状态分布

### 建议在页面中直接展示的字段扩展示例

假设基于平台 `EvidencePackage / DossierDraft / TraceIssue` 领域能力新增 3 个字段：

- `archiveCabinetNo`
- `customerWitness`
- `riskTrendCode`

则页面中应明确展示这些字段如何被落到具体 UI：

- `archiveCabinetNo`
  - 列表：作为“柜号”列显示在证据清单中
  - 新增抽屉：作为卷宗归档表单字段
  - 修改抽屉：可配置为只读或可编辑
- `customerWitness`
  - 查询区：可勾选为筛选条件
  - 新增抽屉：进入替换审批表单
  - 修改抽屉：进入卷宗补录表单
  - 列表：可按角色决定是否显示
- `riskTrendCode`
  - 查询区：可作为风险趋势筛选项
  - 图表：作为 ECharts 维度/系列字段
  - 详情：可在追溯分析说明卡中显示

## 5. 绘制要求

为避免再次画成“说明板”，后续 Pencil 原型应遵守以下规则：

1. 每一页必须像可操作功能页，不能只是说明文字。
2. 每一页至少应出现：
   - 输入框 / 下拉框 / 多选 / 开关 / 表格 / 树 / 按钮 / 状态标签
3. 每一页必须体现当前处于平台链路的哪一步。
4. 为保证演示连贯性，每一页都统一绑定同一个示例上下文：`HJ-2403 / 批次 A-17`。
5. 页面文案以中文为主，仅保留必要术语：
   - `ITP`
   - `ODS/TODS`
   - `ACL`
   - `TraceLink`
   - `Graph Service`
   - `Workflow`
   - `Task Orchestration`
   - `Toolflow`
