# 工业研发底座架构评审报告 (CTO Review)

**评审人：** 首席技术官 (CTO)
**评审对象：** `_bmad-output/planning-artifacts/architecture/` 目录下的统一对象模型、能力地图、技术蓝图、ADR 和前置研究报告。
**日期：** 2026-03-08

---

## 1. 总体评价 (Overall Assessment)

整体而言，这份架构规划**方向极其准确，逻辑非常缜密，且极具实操性**。
从一份“前置研究”推导到完整的“技术蓝图 + 对象模型 + 能力地图 + ADR清单”，很好地把握了当前公司资产（CDM/TDM/MIS/OLTran）的现状，既没有陷入“重写一切全家桶”的技术自嗨，也没有退化为“纯OA式低代码”的浅层拼装。

特别是提出了 **“产品族共用平台内核” (ADR-001)** 的演进模式，以及 **“统一对象优先于统一界面”** 的构建理念，这在工业软件（PLM/SPDM/TDM集成）的复杂现实下，是最落地的最佳实践。我对这份方案的基调非常认可。

---

## 2. 方案首肯亮点 (Top Highlights)

1. **分离但映射的核心对象 (Pragmatic Object Modeling)**
   将 `Project` (偏向CDM组的执行与交付) 与 `ResearchProject` (偏向MIS组的合同与经营) 切分，以及 `Device` (试验台控与物联) 与 `EquipmentAsset` (资产台账与借调) 的解耦，非常符合实体工业企业的实际组织墙，避免了系统大满贯上线时各部门对“一个字段到底谁来填/谁主管”的互撞。
2. **溯源与“数字主线”独立建模 (`TraceLink` as First-class Citizen)**
   明确把 `TraceLink` 及图谱关系单独抽取为核心模型，这是对付如华为DME、达索3DX、易立德等竞品“数字主线 (Digital Thread)”故事的核心护城河和业务刚需点。
3. **清晰的 TDM 边缘架构蓝图与集成策略**
   没有犯“万物皆上中心云”或“纯外围采购代理池”的问题。针对复杂的试验场景和老旧工控，引入了基于 `边缘代理 + 本地预处理 + 断点续传代理` 的模式。这是解决 OT (Operation Technology) 侧海量高频采样、断网降级续传痛点的正确正解。
4. **多模态存储底座与统一模型治理**
   让最合适的库做最适合的事（图数据库供谱系、时序库跑测量、主关系供审批交易）。避免了把海量列式数据/传感器大包硬塞入结构化单库的架构隐患。

## 3. 关键架构缺失与优化建议 (Critical Gaps & Feedback)

作为CTO，从工业软件长远演进、技术壁垒建设与未来商业化的角度判断，我认为有以下几个点需要在**下一迭代的正式版中补强或做出修订**：

### 3.1 工业低代码的边界过窄，应升级为“元模型驱动扩展体系”
**反馈：** 在 ADR-010 架构草案中，方案限制了低/无代码“不进入核心语义层”，初衷对避免业务底座失控是对的。但在真正的工业研发系统中，最大的挑战是频繁的新工装设备型号接入、新采集通道增减、新测量衍生物理量的大量自定义过程。
**建议：** 针对“柔性业务领域对象”（如 TDM 域下的试验规范 `TestSpec`、柔性化测量建模 `MeasurementSchema` 等采集数据模型），**必须放权一套强扩展机制**（Metadata-driven modeling）。
不仅允许前台通过界面重组配置表单，在后台引擎中，也要允许领域专家在“对象注册中心”和规则验证机框架下（受限于权限与审计版本），无需硬编码完成**对象模型的派生与动态宽表/图实体演化**。如果全都要让开发去修改平台内核主键才能上新数据项，这个“底座”会成为企业级创新的开发瓶颈。

### 3.2 CAE 仿真域与算力集群的集成架构（SPDM 与 HPC 场景）偏弱
**反馈：** 在前置研究（`asset-business-model`）中明确提到了目前的资产涉及仿真工具链延伸（`SIM / VLab`）和 OLink 虚实融合等战略资产组件，但在《统一能力地图》与底层蓝图设计中很少看见对于 CAE 仿真计算和调管体系。
**建议：** 工业研发不可避免地涉及超算。目前图上的集成策略是倾向文件级的。需要明确加入针对大算力集群交互架构设计：
- 在 L6 集成与边缘层：建议显性化引入并定格对 **高性能计算集群提交代理网关 (HPC/Job Scheduler Proxy)** 以及**分布式任务节点（Agent）**的管理。并专门针对超大规模仿真结果网格模型（TB级别大文件）在分布式文件存储和云原生对象存储之间，提供“温冷降频迁移/只读虚拟映射”的解法架构处理。

### 3.3 多租户/多法人与部署隔离基线的底层机制缺失
**反馈：** 方案提出了 RBAC+ABAC 组织架构混合模型保障安全管控基线。但考虑到当前公司产品的 SaaS 化可能演化或对大型重工/航天军工集团型国企内网“数据完全物理隔离但流程部分同轨或上下级互通”的部署形态考量（数据主权问题）。
**建议：** 建议明确说明平台是否基于微服务和数据库分表底座原生支持 **逻辑多租户（Logical Multi-tenancy - 租户隔离字段化）** 或者在容器部署配置面支持多命名空间的 **物理多租户隔离架构**。这直接决定了主数据微服务被多站点应用调用时的分发、订阅协议策略以及各层中间件集群的安全设计范式。

### 3.4 “AI赋能/Agent与知识引擎”可以设定得更具侵入性
**反馈：** 方案（ADR-013）将知识与 AI 引擎安全稳当地框定在“侧挂增强”。短期内求稳符合现状。但面对下一代平台技术的发展，“知识反哺协同”将是颠覆传统PLM的重要赛道。
**建议：** 未来建议将大模型与 RAG 能力抽象成标准的“企业专家 Agent 执行器服务”。在工作流引擎（L4 平台）层面不仅是人的审批节点，大模型可以在受控的沙箱隔离状态中作为流程的一个 **“自动化任务节点专家执行器(Worker)”**，由编排器分配执行（如：合同文档预审过滤节点，或交付物标准化自动校验审查过滤节点），审核通过率极大地反哺人工。这属于“流程与AI的融合”，让底座立刻体现出新一代架构的敏捷智能化势能。

## 4. 行动项决议 (Next Steps & ADR Output decisions)

基于上面的讨论，我批准该架构体系进入正式验证与编制阶段，同时请团队针对以下决策做相应动作：

1. **同意并强制冻结以下 P0 架构提议，不再进行大范围重构摇摆：**
   - ADR-001 (基于产品族共用平台内核)
   - ADR-004、ADR-005 (业务闭环与资源账务对象解耦分离，建立统一主数据映射)
   - ADR-006 (TraceLink 数字主线抽离独立建模及治理)
   - ADR-007 (多模混合存储策略确立，抵制粗暴一库大通打)
   - ADR-009 (TDM 实施边-云/中心协同样式定型)

2. **修订补充（待修改重审）架构提议：**
   - **修改 ADR-010 (低/无代码定位)**：除了做前端装配层，加入“强允许对可控的边缘/应用层特化业务对象实现元模型级的无代码柔性扩展机制”的补充规则。这能将业务顾问专家在不发版的情况下解放出来。
   - **补充增发 ADR-015 (仿真协同与异构计算调度 HPC 集成策略)**：确立对于超大文件存储调配，网格解算器和虚拟试验协同在平台与存储、算力中心上的协同通信调度机制。
   - **补充增发 ADR-016 (复杂工业场景部署下的租户分离与数据主权基线方案)**：为未来的上云部署或集团/下沉子公司两级联动部署铺设架构底层保障和数据穿透授权规则体系。

本版本可以基于以上 CTO 建议作为 **Draft v0.2 Release Candidate** 投入后续产品工作拆解。请平台架构师更新你的 ADR 列表，并下达给 CDM / TDM 开发组开展技术细化选型对齐会议。
