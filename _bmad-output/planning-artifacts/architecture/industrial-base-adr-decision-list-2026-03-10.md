1. 先评审 `P0` 决策，不要跳过。
2. 每条 ADR 只能有一个主结论，避免“都要”。
3. 被批准的 ADR 需要反写回对象模型、能力地图和实施路线。
4. 未批准的 ADR 保持 `Proposed` 状态，不进入实施。

## 3. P0 决策项

## ADR-001 底座产品形态

### 决策问题

底座最终应建设成：

- 一个大一统单体超级产品
- 多个独立产品各自演进
- 一个产品族共用的平台内核

### 推荐结论

采用 `产品族共用平台内核`。

### 结论说明

- `CDM / TDM / MIS` 业务语义明显不同，不适合强行收敛为一个单体业务产品。
- 三条产品线在对象模型、权限、文件、流程、集成、运维上存在大量共享能力。
- 平台内核方式既保留产品线差异，又沉淀底座能力。

### 不选方案

- 不选“单一超级产品”，因为会把试验、研发、管理三个域硬揉在一起。
- 不选“完全独立演进”，因为会重复建设共享能力并破坏数字主线。

### 依赖

- 无，最先决策。

## ADR-002 共享内核边界

### 决策问题

哪些能力必须进入共享内核，哪些保留在业务域内？

### 推荐结论

以下能力进入共享内核：

- 统一对象模型
- 身份、组织、角色、字典、主数据
- 文件与对象服务
- 流程引擎、任务引擎
- 搜索、标签、审计
- API 网关、事件总线、文件交换
- 配置中心、发布治理、可观测性

以下能力保留领域：

- CDM 的需求、工作包、评审变更语义
- TDM 的试验建模、采集控制、测量时序处理
- MIS 的合同、设备台账、经营审批语义

### 不选方案

- 不把所有服务都沉到底座，否则底座会膨胀成业务巨无霸。

### 依赖

- ADR-001

## ADR-003 权限与密级模型

### 决策问题

权限是否只做角色控制，还是需要对象级和字段级约束？

### 推荐结论

采用 `RBAC + ABAC + 密级` 混合模型。

### 结论说明

- `RBAC` 解决角色授权。
- `ABAC` 解决组织、项目、对象状态、协同关系等动态授权。
- `密级模型` 负责涉密、导出、跨网、打印、外发等约束。

### 不选方案

- 不选纯 `RBAC`，因为无法覆盖项目制、对象级和密级约束。

### 依赖

- ADR-002

---

## ADR-004 `Project` 与 `ResearchProject` 策略

### 决策问题

研发项目对象和科研管理项目对象是否合并为同一个实体？

### 推荐结论

保留 `Project` 与 `ResearchProject` 双对象，建立统一映射和统一主索引，不强制物理合并。

### 结论说明

- `Project` 更偏研发执行与协同。
- `ResearchProject` 更偏计划、合同、经营和统计。
- 两者业务节奏、状态语义和管理口径不同。

### 不选方案

- 不强制一表化，否则会把管理语义和研发语义互相污染。

### 依赖

- ADR-002
- ADR-006

---

## ADR-005 `Device` 与 `EquipmentAsset` 策略

### 决策问题

试验设备对象和资产设备对象是否统一为一个对象？

### 推荐结论

建立统一设备主索引，`Device` 与 `EquipmentAsset` 分域建模、双向映射。

### 结论说明

- `Device` 面向试验执行、采集控制、站点接入。
- `EquipmentAsset` 面向资产台账、借调、维修、报废。
- 两者共享基础标识和分类，但不共享全部属性。

### 不选方案

- 不做单一大对象，避免资产属性和试验属性混杂。

### 依赖

- ADR-002

---

## ADR-006 数字主线建模策略

### 决策问题

对象之间的主线与谱系关系，是做成普通关系表，还是独立能力？

### 推荐结论

`TraceLink` 作为一级对象，并形成独立的主线/谱系服务。

### 结论说明

- 研发、试验、管理三条链需要跨域追溯。
- 关系类型不只是外键，还包括来源、验证、派生、影响、替代、归档等。
- 需要审计、版本化、可查询和可视化能力。

### 不选方案

- 不做普通业务表附属字段，否则后期无法支撑数字主线能力。

### 依赖

- ADR-002
- ADR-004
- ADR-005

## ADR-007 存储架构策略

### 决策问题

底座是否应该用单一数据库统一承载全部对象与数据？

### 推荐结论

采用 `多模存储矩阵`：

- 关系库：主数据、事务数据、审批数据
- 文档库：柔性对象、扩展属性
- 对象存储：文件、图纸、报告、模型包
- 时序库/列式库：采集和测量数据
- 图数据库：主线、谱系、知识网络
- 搜索引擎：全文检索、标签、导航

### 不选方案

- 不选单库统管，尤其不把时序数据、图谱关系和大文件强塞进同一引擎。

### 依赖

- ADR-006

---

## ADR-008 流程/任务引擎策略

### 决策问题

流程和任务能力是由各业务域各自建设，还是平台统一提供？

### 推荐结论

流程引擎和任务引擎由平台统一提供，业务域只负责场景模板和编排逻辑。

### 结论说明

- 现有资产已经体现出工作流、任务流、工具流、数据流多引擎特征。
- `MIS/SUNBPM` 的可复用部分应沉为平台能力，而不是要求所有产品依赖完整 MIS 形态。
- 业务域只做语义增强和模板化。

### 不选方案

- 不允许各域自行建设独立流程引擎，否则审批、待办、通知和审计无法统一。

### 依赖

- ADR-001
- ADR-002

## ADR-009 TDM 边缘与中心边界

### 决策问题

试验数据接入应全部中心化，还是中心加边缘协同？

### 推荐结论

采用 `中心平台 + 边缘代理 + 本地缓存 + 异步同步` 模式。

### 结论说明

- 试验站点网络环境和设备接入条件差异大。
- 大文件和测量数据不适合全部实时直传中心。
- 跨网、断网和边缘预处理都是刚需。

### 不选方案

- 不选纯中心化接入，否则站点稳定性、时延和跨网问题会持续放大。

### 依赖

- ADR-007
- ADR-012

## ADR-010 低/无代码定位

### 决策问题

低/无代码能力是底座主轴，还是受控装配层？

### 推荐结论

定位为 `受控装配层 + 元模型驱动扩展体系`，并纳入 `L1-L4` 四层扩展治理框架。

### 允许承接

- 门户、导航、工作台
- 表单、查询、看板
- 审批流、通知流
- 轻量规则
- 轻量集成编排
- 白名单柔性领域对象派生，如 `TestSpec`、`MeasurementSchema` 的受控扩展

### 扩展前提

- 必须挂靠统一对象注册中心
- 必须有模型版本、审批、发布和回滚
- 必须明确位于 `L4 场景装配 / L3 领域模型扩展 / L2 平台能力扩展 / L1 内核` 中的哪一层
- 不得绕开权限、密级、审计和基线机制
- 仅允许在白名单对象范围内做派生和字段演化

### 禁止承接

- 核心对象定义
- 核心权限与密级
- 核心版本和基线
- 深度工程集成
- 大规模测量数据处理

### 不选方案

- 不选“通用低代码平台优先”，否则底座会退化为 OA/PaaS。

### 依赖

- ADR-002
- ADR-003
- ADR-006

## ADR-020 MBSE/架构对象建模策略

### 决策问题

工业研发底座是否只管理 `Requirement / Project / Task / DataSet` 等执行与数据对象，还是需要把 `功能架构 / 逻辑架构 / 物理架构 / 接口架构` 提升为一级对象？

### 推荐结论

将 `ArchitectureDefinition / FunctionArchitecture / LogicalArchitecture / PhysicalArchitecture / InterfaceSpec / VerificationCase` 提升为一级对象与一级服务。

### 结论说明

- 架构对象是 `需求 -> 设计/仿真/试验` 之间的承上启下层。
- 如果不单独建模，`TraceLink` 会缺失最关键的数字主线段。
- 行业竞品与自有协同研发资料都证明“架构驱动研发”是工业软件的关键壁垒之一。

### 不选方案

- 不把架构对象退化成 `Deliverable` 或 `KnowledgeAsset` 附件。
- 不把架构对象只做成工具侧临时文件映射。

### 依赖

- ADR-002
- ADR-006
- ADR-008

## ADR-021 TDM 标准语义兼容策略

### 决策问题

试验域是只保留业务侧 `MeasurementSchema`，还是需要同时建设 `ODS/TODS` 标准语义兼容层？

### 推荐结论

采用双轨建模：

- `MeasurementSchema` 承载业务侧采集、通道和执行语义
- `TODSApplicationModel` 承载标准侧语义、交换和接口兼容
- 两者通过映射关系关联，而不是互相替代

### 结论说明

- `MeasurementSchema` 解决“业务建模”问题。
- `TODS/ODS` 解决“标准兼容、跨系统交换、语义互通”问题。
- 只有双轨建模，底座才能既适配当前业务，又保留标准化扩展能力。

### 不选方案

- 不把 `MeasurementSchema` 直接等同于 `TODSApplicationModel`。
- 不把 `ODS/TODS` 降格成附件规范或接口备注。

### 依赖

- ADR-007
- ADR-009
- ADR-011

## ADR-012 安全与保密架构

### 决策问题

安全是否只做平台登录和接口鉴权，还是深入到对象、字段、导出、跨网链路？

### 推荐结论

采用 `对象级 + 字段级 + 行为级 + 跨网链路级` 的统一保密模型。

### 结论说明

- 需要覆盖对象读取、对象变更、文件下载、导出、打印、外发、跨网传输。
- 安全模型必须进入对象、流程、文件、边缘接入和低代码发布链路。

### 不选方案

- 不选“外围网关安全”模式，因为无法覆盖实际业务对象行为。

### 依赖

- ADR-003
- ADR-007
- ADR-009
- ADR-010

## ADR-015 租户与部署隔离策略

### 决策问题

平台是否需要同时支持逻辑多租户、物理隔离部署和多法人组织模式？

### 推荐结论

支持三种标准化模式：

1. 单法人单实例
2. 多法人逻辑隔离
3. 高保密客户物理隔离部署

### 结论说明

- 不同客户的安全与部署诉求差异极大，不能只用一种模式覆盖。
- 集团型客户常要求“流程同轨，数据隔离”。
- 军工/涉密客户通常要求物理边界与独立运维。

### 标准化要求

- 服务隔离策略标准化
- 数据库隔离策略标准化
- 对象存储隔离策略标准化
- 搜索、缓存、消息隔离策略标准化
- 主数据分发和订阅策略标准化

### 不选方案

- 不选“只支持一种部署模型”，否则会在交付阶段反复定制架构。

### 依赖

- ADR-003
- ADR-012

## ADR-017 行业模板包（ITP）标准

### 决策问题

平台是否需要一个标准化、可版本化、可加载的行业交付载体来承接快速复制？

### 推荐结论

采用 `Industry Template Package (ITP)` 作为快速复制交付、租户初始化和行业方案装配的标准载体。

### 结论说明

- `ITP` 统一封装行业模型、流程、视图、规则、主数据初始化、连接器配置和权限模板。
- 模板包必须具备 `manifest.yaml`、版本、依赖、兼容性和插件挂载点。
- 模板包加载必须走统一发布链路，并与 `Tenant / SchemaVersion / ReleasePackage` 建立映射。

### 不选方案

- 不选“项目交付材料零散拼装”，否则无法形成可复制的交付引擎。
- 不选“模板只停留在表单和流程层”，否则无法承载工业场景的行业复用。

### 依赖

- ADR-002
- ADR-010
- ADR-015

## ADR-014 演进路线策略

### 决策问题

底座建设应先做什么，后做什么？

### 推荐结论

按以下顺序推进：

1. 统一对象模型和共享内核边界
2. 建立 `ITP + MetaObjectType + SchemaVersion + 租户初始化` 复制引擎
3. 统一权限、文件、流程、任务、集成和审计能力
4. 统一试验接入、HPC 协同和数据治理
5. 再做低/无代码装配、模板资产和模型自动联动
6. 最后做知识图谱、语义检索、AI Worker 和生态开放

### 不选方案

- 不选“先做统一门户和大屏”，那样只统一了表面。

### 依赖

- 依赖全部 P0 决策。

## ADR-018 模型驱动自动联动策略

### 决策问题

`MetaObjectType / SchemaVersion` 发布后，平台是一步到位生成完整 API，还是分阶段实现自动联动？

### 推荐结论

采用两阶段策略：

1. `Phase 1`：发布后自动联动注册搜索索引、权限点、审计类别、基础视图和事件订阅
2. `Phase 2`：在治理成熟后，再进入模型驱动 API 自动生成

### 结论说明

- 先做自动联动可以快速产生平台价值，又不会过早把运行时复杂度推高。
- 直接承诺“注册即生效 + 全量 API 自动生成”会放大治理、兼容性和安全风险。
- `SchemaVersion` 变更通知必须能够驱动下游消费者同步更新。

### 不选方案

- 不选“全部手工注册”，否则元模型就无法转化为复制效率。
- 不选“一步到位全自动生成”，否则会把底座提前推入高复杂运行时。

### 依赖

- ADR-010
- ADR-017

## ADR-019 领域对象存储实现策略

### 决策问题

核心对象和柔性对象是否采用同一种持久化方式，还是采用混合存储策略？

### 推荐结论

采用 `稳定关系模型 + JSON/文档混合模型` 的分层策略：

- 核心对象使用稳定关系模型，可辅以扩展表
- 柔性领域对象使用 JSON 字段或文档模型承接宽表和快速演化
- 所有对象统一纳入 `MetaObjectType` 与对象注册中心治理

### 结论说明

- 核心对象更关注事务一致性、审计和关联查询。
- 柔性对象更关注模型演化速度和模板复制效率。
- 统一注册优先于统一物理存储。

### 不选方案

- 不选“全量 EAV”，查询与报表代价过高。
- 不选“全部固定表继承”，柔性对象演化会把交付速度拖回定制开发。

### 依赖

- ADR-007
- ADR-010
- ADR-017

## ADR-022 工程工作台与 B/C/S 通道策略

### 决策问题

平台是否统一收敛为纯浏览器通道，还是同时保留 `Web Portal` 与 `Engineering Workbench` 两类入口？

### 推荐结论

平台同时支持：

- `Web Portal`：门户、工作台、查询、审批、轻量装配
- `Engineering Workbench`：CAD/CAE/仿真、工具流、模型轻量化、复杂交互

并把 `B/S + B/C/S` 视为正式架构通道，而不是历史兼容细节。

### 结论说明

- 工业研发场景存在大量复杂交互、工具会话和模型浏览需求。
- 纯 Web 通道会压缩工程工作场景，迫使专业能力回到私有客户端或离线工具。
- 平台治理仍然统一，包括会话、审计、任务回传、凭据和结果回挂。

### 不选方案

- 不强制所有工程场景退化为纯浏览器模式。
- 不允许 Workbench 脱离平台审计、任务和权限体系独立演进。

### 依赖

- ADR-008
- ADR-011
- ADR-012

## ADR-023 统一 Graph Service 策略

### 决策问题

模型依赖图谱、实例 `TraceLink` 主线图谱和后续知识图谱，是做成多套独立图服务，还是统一到一个图服务体系？

### 推荐结论

采用 **统一 `Graph Service` 逻辑接口，底层按图域和负载分治** 的策略：

- 对上统一查询、遍历、影响分析和治理语义
- 对内允许模型图、实例图、知识图按后端和容量独立演进
- `Trace Service` 继续作为实例 `TraceLink` 的写入权威

### 结论说明

- 模型依赖图谱和实例主线图谱本质上都是图关系管理问题，统一服务语义更利于影响分析和主线解释一致。
- 若完全拆成多套服务，后续会形成两套关系语义和两套治理口径。
- 若强行压到一个物理图库，又会放大写入、容量、冷热分层和隔离风险。
- 统一应落在逻辑服务层，分治应落在投影和存储实现层。

### 不选方案

- 不选“模型图和实例图完全分开做两套服务”，避免语义分叉。
- 不选“所有图能力强制共用一个物理图库”，避免把逻辑统一误解成物理统一。
- 不允许页面、脚本、插件直接写图数据库。

### 依赖

- ADR-006
- ADR-007
- ADR-017
- ADR-018

## ADR-024 平台 UI 架构策略

### 决策问题

平台 UI 层应采用什么架构模式来同时满足：
- 复杂交互能力（图形化拖拽、连线、多层嵌套工作台）的品质要求
- 灵活可配置（字段增删、组件挂载、布局调整）的扩展要求
- NocoBase 式"运行即配置"的所见即所得编辑体验
- 工业场景下的治理约束（变更审批、影响分析、兼容性校验）

### 推荐结论

采用 `引擎 + Schema 混合模式`：

- **引擎层**：硬编码，保证复杂交互品质（graph-canvas、form-engine、table-engine 等）
- **能力组件层**：硬编码 + 配置接口（io-configurator、relation-browser 等）
- **页面组合层**：声明式 UI Schema，编辑态可原地修改

运行态和编辑态共享同一份 UI Schema，由**双模渲染引擎**驱动切换。编辑态通过 `ExtensionPoint` 控制可编辑边界，变更通过 `DesignChangeset` 暂存并经影响分析后发布。

### 结论说明

- 纯 Schema 驱动（如 NocoBase）无法承载工业场景下的复杂交互（图形化工作包分解、输入输出配置器、使能/控制关系面板等），必须有硬编码引擎层保证品质。
- 纯硬编码无法满足"实施顾问自行扩展字段和页签"的灵活性要求，必须有 Schema 层暴露配置面。
- 混合模式兼顾两者：引擎保证交互品质上限，Schema 保证灵活扩展下限。
- `ExtensionPoint` 机制实现"受控灵活"——M1 层预埋可扩展点，M2 层在预埋点内扩展，不是无限灵活。
- `DesignChangeset` 的 Draft → 影响分析 → 发布流程，保障工业场景"不能随便改生产系统"的治理要求。
- "基于领域产品扩展"和"从0设计"两种模式的产出物同构——都是同一元模型的 UI Schema + ExtensionPoints，进入同一注册与治理管线。

### 不选方案

- 不选"纯 Schema 驱动"（如 NocoBase 原生模式），因为无法承载 graph-canvas 等复杂引擎级交互。
- 不选"纯硬编码 + 配置文件"，因为无法支持编辑态的所见即所得体验和实施顾问自行扩展。
- 不选"独立设计器 + 独立运行时"，因为两套系统的一致性维护成本过高，且 Feature Wizard 向导式体验不如原地编辑直观。

### 关键约束

1. 引擎层和能力组件通过代码发布更新，不走 Schema
2. Schema 编辑态仅操作 Schema 层，不触及引擎层代码
3. 编辑态变更必须经 `DesignChangeset` 的 Draft → 影响分析 → 发布流程
4. `FeatureExtension` 是编辑态操作的自动产出物，不是用户手写的输入
5. 引擎库的丰富度决定平台能力边界，建议分批建设

### 依赖

- ADR-010（低/无代码定位）
- ADR-017（ITP 标准）
- ADR-022（工程工作台与通道策略）

### 影响

- 统一对象模型 §13.5 新增 `ExtensionPoint`、`FeatureExtension`、`DesignChangeset`、`ComponentType`
- 统一对象模型 §13.5.9-13.5.12 新增能力包 FED 控制机制、包级编排模块、FED 晋升治理、P0-P5 完整层次结构
- 统一对象模型 §13.5.13-13.5.14 新增 Extension Unit（ServerUnit/ClientUnit），废弃原 plugins 概念，ExtensionPoint 范围扩展至后端层，FED 增加 orchestration 编排声明和 extensionUnits 引用，ITP 包结构 `behavior/plugins/` 废弃改为 `behavior/server-units/` + `behavior/client-units/`
- ITP 包结构 `behavior/views/` 从"视图模板引用"演进为"完整 UI Schema"
- ITP 包结构新增 `pack-coordination/` 目录承载能力包级声明和 FED 协调
- 包级编排模块（OrchestrationModule）正式纳入 P1 能力包层标准概念
