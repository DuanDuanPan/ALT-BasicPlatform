# 工业研发底座四层扩展治理体系

状态：`Draft v0.1`  
日期：`2026-03-08`  
用途：把当前底座的扩展能力从分散规则，收敛为统一的四层治理框架。

---

## 1. 目标

这份治理体系解决三件事：

1. 不同角色到底能改到哪一层
2. 哪些扩展要审批、版本化、回滚
3. 如何防止“低代码侵入内核”或“平台开发绕开治理”

---

## 2. 四层扩展模型

```text
L4 场景装配层
L3 领域模型扩展层
L2 平台能力扩展层
L1 内核层
```

---

## 3. 分层定义

## 3.1 L1 内核层

定位：

- 平台不可扩展核心
- 只能升级，不能被业务方重定义

包含内容：

- 核心对象语义
- 权限/密级/审计模型
- `TraceLink` 主线引擎
- 租户隔离与安全基础设施

操作主体：

- 平台核心团队

治理要求：

- 必须架构评审
- 必须核心版本管理
- 仅限受控升级

## 3.2 L2 平台能力扩展层

定位：

- 平台级通用扩展能力
- 需要开发介入，但不能破坏内核

包含内容：

- 连接器开发和注册
- 规则引擎脚本
- AI Worker 定义和挂载
- 自定义服务 API（通过 SDK）

操作主体：

- 平台开发者

治理要求：

- 必须发布审批
- 必须 `ReleasePackage` 版本化
- 必须支持回滚

## 3.3 L3 领域模型扩展层

定位：

- 白名单柔性对象的受控扩展
- 这是快速复制和行业差异化的核心层

包含内容：

- 白名单对象属性扩展
- 白名单对象派生定义
- 行业模板包加载与版本化
- 模型版本化、审批、发布、回滚

操作主体：

- 实施架构师
- 领域架构师

治理要求：

- 必须模型审批流
- 必须 `SchemaVersion` 版本化
- 必须支持差量追踪和回滚

## 3.4 L4 场景装配层

定位：

- 业务场景配置层
- 面向实施顾问和客户管理员

包含内容：

- 页面布局
- 字段排序
- 看板配置
- 查询条件
- 流程模板选择
- 节点角色配置

操作主体：

- 实施顾问
- 客户管理员

治理要求：

- 本地配置可快速生效
- 仍需纳入版本管理和回滚
- 不允许越界到 L3/L2/L1

---

## 4. 准入矩阵

| 层级 | 谁能操作 | 需要审批 | 版本化 | 可回滚 | 典型对象 |
|---|---|---|---|---|---|
| L4 场景装配 | 实施顾问、客户管理员 | 否 | 是 | 是 | ViewTemplate、DashboardView |
| L3 领域模型扩展 | 实施架构师、领域架构师 | 是 | 是 | 是 | MetaObjectType、SchemaVersion |
| L2 平台能力扩展 | 平台开发者 | 是 | 是 | 是 | ReleasePackage、Connector、AIWorkerTask |
| L1 内核 | 平台核心团队 | 是 | 是 | 受限 | 核心对象、Policy、TraceLink |

---

## 5. 层间边界

### L4 不能做什么

- 不能新增核心对象
- 不能修改权限/密级模型
- 不能修改审计链路
- 不能修改租户隔离策略

### L3 不能做什么

- 不能重定义核心对象语义
- 不能绕开发布闸门
- 不能直接改核心服务代码

### L2 不能做什么

- 不能通过插件绕过权限和审计
- 不能注入跨租户访问逻辑

### L1 的升级限制

- 只能通过核心版本升级
- 必须附带兼容性说明和迁移说明

---

## 6. 治理流程

## 6.1 L4 流程

配置 -> 本地校验 -> 保存版本 -> 发布到场景 -> 回滚可用

## 6.2 L3 流程

模型变更申请 -> 审批 -> SchemaVersion 生成 -> 兼容性校验 -> 发布 -> 审计记录

## 6.3 L2 流程

扩展开发 -> 安全扫描 -> 发布审批 -> ReleasePackage 发布 -> 挂载 -> 回滚能力验证

## 6.4 L1 流程

架构评审 -> 兼容性评审 -> 版本冻结 -> 发布窗口 -> 升级验证

## 8. 推荐落地顺序

### 第一批

- 先落 L4 场景装配边界
- 同时落 L3 模型审批流

### 第二批

- 再落 L2 连接器/AI Worker 扩展发布机制

### 第三批

- 最后完善 L1 升级兼容策略

---

## 9. 当前结论

如果没有这套四层治理体系，底座后续一定会在两个极端间摆动：

- 要么什么都不敢开放，复制能力上不去
- 要么什么都能配，最后把内核配烂

四层治理体系的作用，就是把“开放”变成“受控开放”。
