# ADR-023 统一 Graph Service：逻辑统一、底层分治

状态：`Proposed / Draft v0.1`  
日期：`2026-03-12`  
优先级：`P1`

---

## 1. 背景

当前 `v0.5` 架构材料已经形成以下共识：

- `TraceLink` 是一级对象，必须由独立 `Trace Service` 管理，而不是退化为普通关系表。
- 模型级依赖图谱和实例级 `TraceLink` 图谱，本质上都是图关系管理和图遍历问题。
- 图数据库、搜索和知识索引都属于投影，不承载业务事务真相。
- `L3` 领域服务不允许直接写图数据库；页面装配层、脚本和插件也不允许绕过服务边界直接写入。

但目前尚缺一条独立 ADR，回答以下问题：

1. 模型依赖图谱和实例主线图谱，是做两套服务，还是统一为一个图服务体系？
2. `Graph Service` 与 `Trace Service`、`Object Model Registry`、知识投影器之间的边界如何划分？
3. 是否要求所有图能力落在同一个物理图库，还是允许按域分治？

---

## 2. 决策

采用 **路线 B：统一 Graph Service，底层可分治**。

即：

- 对上提供统一的 `Graph Service` 逻辑接口和治理语义；
- 对内按图域和负载特征对底层投影与存储进行分治；
- `Trace Service` 继续作为实例级 `TraceLink` 的写入权威；
- 模型依赖图谱、实例主线图谱、知识图谱等图域可使用同一服务语义层，但不强制共用同一物理图库或同一存储拓扑。

---

## 3. 决策范围

本 ADR 约束以下内容：

- 模型依赖图谱
- 实例级 `TraceLink` 主线图谱
- 影响分析与图遍历服务
- 图查询接口和服务职责边界
- 图投影的物理部署与演进策略

本 ADR 不直接决定：

- 图数据库具体选型
- 知识图谱的本体设计细节
- 搜索引擎的字段结构
- 具体 UI 展现方式

---

## 4. 推荐架构

### 4.1 统一的是逻辑服务，不是单一物理库

`Graph Service` 的统一，指的是：

- 统一图节点/边语义抽象
- 统一图查询和遍历接口
- 统一租户、权限、密级和审计接入规则
- 统一影响分析、路径查询、邻域查询、子图查询、差异查询的能力入口

不要求：

- 所有图域共用一个物理图库
- 所有图数据共用一个集群
- 所有图查询共用同一索引结构

### 4.2 图域划分

建议至少区分以下三个逻辑图域：

1. **模型图域**
   - 节点：`SchemaVersion`、对象模型、模板、连接器声明、合规声明、视图模板等
   - 边：依赖、引用、包含、绑定、覆盖
   - 用途：装载校验、裁剪影响分析、冲突检测、依赖解释

2. **实例图域**
   - 节点：运行时对象实例
   - 边：`TraceLink`
   - 用途：正向追踪、反向溯源、变更影响、主线视图

3. **知识图域**
   - 节点：知识条目、标准项、经验资产、术语
   - 边：引用、归类、映射、相似、来源
   - 用途：知识导航、推荐、智能检索增强

`Phase 0 / v0.5` 最小闭环优先落前两域；知识图域可后续接入同一 `Graph Service` 语义层。

### 4.3 服务边界

#### `Trace Service`

负责：

- 接收 `TraceIntent`
- 校验关系类型和写入规则
- 幂等创建或更新 `TraceLink`
- 失败补偿、重试和失效标记

不负责：

- 通用图遍历 API 的统一对外暴露
- 模型依赖图谱的扫描与解释

#### `Graph Service`

负责：

- 提供统一图查询/遍历/影响分析接口
- 路由到模型图域、实例图域、知识图域的底层投影
- 统一做权限、租户、密级过滤
- 统一封装路径查询、邻域查询、子图查询、差异查询和解释输出

不负责：

- 承载业务事务真相
- 直接接收页面、脚本、插件写入图数据库
- 替代 `Trace Service` 的实例关系写入责任

#### `Object Model Registry / Model Automation`

负责：

- 在 `SchemaVersion` 发布或 `ITP` 装载时解析模型依赖
- 产出模型图域投影和影响分析视图的注册输入

#### 知识投影器

负责：

- 将知识资产关系投影到知识图域
- 不绕过统一图服务治理边界

---

## 5. 写入与查询责任

### 5.1 写入责任

采用“分域写入、统一查询”的原则：

- 模型图域：由 `Meta-model Extension / Model Automation / ITP Loader` 负责投影写入
- 实例图域：由 `Trace Service` 负责写入
- 知识图域：由知识投影器负责写入

禁止：

- `L3` 领域服务直接写图数据库
- 页面装配层、脚本、插件直接写图数据库
- 通过自定义插件修改 `TraceLink` 核心写入逻辑

### 5.2 查询责任

统一由 `Graph Service` 对外提供以下查询能力：

- 依赖分析
- 影响分析
- 正向追踪
- 反向溯源
- 多跳路径查询
- 子图导出
- 视角化图解释

若某个场景只需要简单读模型或对象列表，不强制通过 `Graph Service`，仍可走对象服务、搜索服务或读投影。

---

## 6. 底层分治策略

### 6.1 分治原则

底层存储与投影允许按以下维度分治：

- 图域分治：模型图、实例图、知识图分别演进
- 负载分治：高写入主线图与高查询模型图分开
- 隔离分治：不同租户或涉密环境采用命名图、子图或独立实例
- 生命周期分治：热图、温图、归档图分层

### 6.2 可接受形态

以下形态均可接受：

1. **统一服务 + 单图存储实例 + 多命名图**
   - 适合早期验证和小规模部署

2. **统一服务 + 模型图/实例图分库**
   - 适合 `TraceLink` 写入量明显高于模型图时

3. **统一服务 + 图数据库 + 关系投影/搜索投影混合**
   - 适合部分模型影响分析先由关系投影或索引支持

4. **统一服务 + 多图存储后端**
   - 适合中心、边缘、涉密环境分治部署

### 6.3 不接受形态

以下形态不接受：

1. 两套完全独立、彼此语义不一致的图服务体系
2. 所有图能力强制压到一个物理图库，且不区分负载和生命周期
3. 页面、插件、脚本绕开统一边界直写图投影
4. 让图数据库承载业务事务真相

---

## 7. 选择路线 B 的原因

### 7.1 相比路线 A：完全分开做

路线 A 的问题：

- 模型图谱和实例图谱会形成两套关系语义
- 查询接口和治理口径容易分叉
- 影响分析、主线追踪和知识关系难以统一解释
- 后续会出现“图数据库统一了，服务语义却不统一”的二次整合成本

路线 B 的优势：

- 先统一语义层和服务契约
- 保留底层分治空间
- 既能支撑 `v0.5` 快速落地，又不锁死后续演进

### 7.2 相比路线 C：单一物理图库统管一切

路线 C 的问题：

- 模型图和实例图在写入频率、数据规模、冷热特征上差异很大
- 一开始就绑定同一物理拓扑，会放大容量、隔离、归档和性能风险
- 容易把“逻辑统一”误解为“物理统一”

路线 B 的优势：

- 把统一放在正确层级：服务语义和治理语义
- 把分治放在正确层级：投影和存储实现

---

## 8. 一致性与重建策略

### 8.1 一致性原则

- 图谱是投影，不是业务主真相
- 图谱更新采用最终一致
- 业务主事务不等待图谱写入完成
- 图谱失败进入补偿队列，不直接回滚业务主事务

### 8.2 重建原则

任一图域都必须支持：

- 全量重建
- 增量修复
- 版本失效标记
- 按租户/按 `SchemaVersion`/按 `ITP` 粒度重建

---

## 9. 安全与治理要求

`Graph Service` 必须继承统一治理要求：

- 租户隔离
- `RBAC + ABAC + 密级` 过滤
- 图查询审计
- 导出审批控制
- 跨网与涉密环境约束

禁止把图查询做成绕过权限模型的“技术后门”。

---

## 10. 最小实施建议

### Phase 0

- 统一定义 `Graph Service` 接口语义
- 实现模型图域 + 实例图域最小闭环
- `Trace Service` 保持实例图写入权威
- 提供最少一组能力：依赖分析、正向追踪、反向溯源、变更影响

### Phase 1

- 按负载把模型图和实例图投影拆分
- 引入图查询解释能力和视角化输出
- 接入知识图域

### Phase 2

- 支持多后端路由
- 支持按隔离模式切换物理部署
- 支持图谱重建编排和容量治理自动化

---

## 11. 验收标准

采纳本 ADR 后，至少应满足：

1. `TraceLink` 仍只允许通过 `Trace Service` 写入。
2. 模型依赖分析和实例主线查询从统一 `Graph Service` 暴露。
3. 图存储可按图域或负载拆分，而不影响上层接口。
4. `Graph Service` 能统一执行租户、权限和密级过滤。
5. 任一图域支持补偿、失效标记和重建。
