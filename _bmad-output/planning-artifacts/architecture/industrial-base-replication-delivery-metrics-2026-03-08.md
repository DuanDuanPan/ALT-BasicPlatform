# 工业研发底座复制交付指标与 ROI 矩阵

状态：`Draft v0.1`  
日期：`2026-03-08`  
用途：将“快速复制交付”从战略口号拆成可追踪的交付指标、复用率矩阵和 ROI 假设。

---

## 1. 目标

这份指标矩阵回答三个问题：

1. 什么叫“复制成功”
2. 哪些指标能证明底座真的在降本增效
3. 什么时候共享服务值得进入第一批

---

## 2. 核心目标值

建议以以下目标作为 `Phase 1` 口径：

| 指标 | 当前模式 | 目标模式 |
|---|---|---|
| 新客户交付周期 | 20-24 周 | 3-4 周 |
| 单项目交付总人月 | 23-28 人月 | 5-7 人月 |
| 核心配置比例 | <20% | >60% |
| 样板模板复用率 | <10% | >70% |
| 标准初始化自动化率 | <10% | >80% |

说明：

- 这些是目标值，不是当前承诺值
- 正式版应再补敏感性区间

---

## 3. 交付漏斗指标

建议跟踪从签约到上线的全过程：

| 阶段 | 指标 | 目标 |
|---|---|---|
| 模板选择 | 选型耗时 | `<= 1 天` |
| 租户初始化 | 初始化耗时 | `<= 1 天` |
| 主数据配置 | 初始配置耗时 | `<= 3 天` |
| 流程配置 | 流程可用耗时 | `<= 3 天` |
| 页面调整 | 基础页面就绪耗时 | `<= 3 天` |
| 连接器配置 | 标准连接器就绪耗时 | `<= 5 天` |
| 联调 | 端到端联调耗时 | `<= 5 天` |
| 验收 | 初次验收耗时 | `<= 3 天` |

---

## 4. 复用率矩阵

建议共享能力进入第一批的门槛：

> 预估复用率 `>= 60%`

| 能力 | 复用对象 | 预估复用率 | 结论 |
|---|---|---|---|
| Identity Service | CDM/TDM/MIS 全部项目 | >90% | 第一批 |
| Master Data Service | 全部项目 | >85% | 第一批 |
| Workflow Service | 大多数项目 | >80% | 第一批 |
| File/Object Service | 多数项目 | >75% | 第一批 |
| Search Service | 多数项目 | >70% | 第一批 |
| Trace Service | 研发/试验主线项目 | 50-70% | 第一批边缘 |
| AI Worker | 高级场景 | <40% | 第二批 |

---

## 5. 成本收益假设

基于 Party Mode 报告的估算，可先采用以下假设模型：

| 指标 | 假设值 |
|---|---|
| 年新客户数 | 8 |
| 当前单项目人月 | 23-28 |
| 模板模式单项目人月 | 5-7 |
| 人月成本 | 3 万 |
| 年节省人月 | ~150 |
| 年节省成本 | ~450 万 |

补充要求：

- 正式版应区分“最佳/中性/保守”三种情景
- 不直接把此值作为财务承诺

---

## 6. 用户旅程口径

建议至少围绕一个标准旅程评估交付效率：

```text
签约 -> 选择 ITP -> 创建租户 -> 加载模板 -> 注入主数据
    -> 配置流程 -> 调整页面 -> 配置连接器 -> 联调 -> 验收
```

这个旅程应成为：

- 交付团队的标准流程
- 销售对外承诺的边界
- 平台团队验证复制能力的共同基准

---

## 7. 领先指标与滞后指标

## 7.1 领先指标

- ITP 加载成功率
- 模板版本复用率
- 白名单对象扩展成功率
- 初始化自动化完成率
- 标准连接器复用率

## 7.2 滞后指标

- 单项目实际交付人月
- 上线周期
- 交付返工率
- 客户个性化开发占比

---

## 8. 当前结论

如果没有这套指标矩阵，底座建设会出现两个问题：

- 技术上觉得在进步，但业务看不到价值
- 所有人都说“复用很多”，但没有统一口径

复制交付指标的作用，就是把“平台价值”从叙述变成数字。
