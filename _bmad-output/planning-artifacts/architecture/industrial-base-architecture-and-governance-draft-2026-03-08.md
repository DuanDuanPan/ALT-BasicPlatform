# 工业研发底座架构设计与治理草案

状态：`Draft v0.3`  
日期：`2026-03-08`  
定位：基于现有资产盘点与业务建模结果形成的前置架构草案，用于指导后续《底座架构设计与治理》正式版编制。  
说明：当前目录下未发现正式 `PRD`，因此本文件是 `研究驱动型架构草案`，不是需求冻结后的正式架构决策文档。

---

## 1. 文档目的

本草案回答四个问题：

1. 现有 `CDM + TDM + 管理信息化` 资产，应被抽象成什么样的底座目标。
2. 底座应该按哪些业务域、能力域、数据域和治理域来设计。
3. 低/无代码能力是否需要纳入，以及应该放在什么位置、管到什么边界。
4. 下一阶段应按什么顺序推进，避免平台化失焦。

---

## 2. 执行摘要

### 2.1 总体判断

建议将底座定义为：

> 面向工业研发场景的统一对象模型与统一协同执行平台，向上同时支撑协同研发、数字化试验、管理信息化，并通过数据、模型、知识和集成能力沉淀为企业级工业研发底座。

这一定义意味着底座不是：

- 一个通用 BPM/OA 平台
- 一个单纯的数据中台
- 一个只服务研发协同的项目系统
- 一个只做表单和流程装配的低代码平台

这一定义意味着底座必须同时具备：

- `对象中心`：统一对象模型、主数据、版本、状态、基线、密级
- `执行中心`：流程流转、任务协同、工具调用、数据处理
- `数据中心`：文件、对象、图谱、测量/时序数据统一纳管
- `集成中心`：IT 系统、OT 设备、外部工具、工业 APP 统一接入
- `知识中心`：知识库、图谱、语义检索、智能辅助
- `复制中心`：ITP、SchemaVersion、初始化流程、自动联动、模板资产
- `治理中心`：权限、安全、审计、发布、配置、运维、国产化

### 2.2 核心结论

1. 底座应围绕 `统一对象模型 + 多引擎协同 + 数据治理 + 集成治理` 展开，而不是先做界面统一。
2. `低/无代码能力` 需要补，但应定位为 `工业模型驱动装配层 + 元模型驱动扩展体系`，而不是通用平台的主轴。
3. `CDM` 负责研发协同语义，`TDM` 负责试验与数据语义，`MIS/SUNBPM` 负责经营与主数据语义，三者要收敛到同一平台核心。
4. 从竞品对比看，未来壁垒不在单独流程能力，而在 `对象模型 + 数字主线 + 试验/仿真/协同一体化 + 强治理`。
5. 底座必须内生 `ITP + SchemaVersion + 租户初始化 + 发布后自动联动` 的复制引擎，否则仍会滑回项目制交付。

## 4. 目标定位与设计原则

## 4.1 目标定位

底座的最终形态应是一个 `工业研发统一底座`，而不是多个独立产品的简单拼装。其职责包括：

- 统一业务对象定义和跨系统标识
- 统一流程、任务、工具、数据之间的运行机制
- 统一文件、对象、图谱、测量数据的治理规则
- 统一外部系统与设备的接入规范
- 统一权限、密级、审计、版本、发布和运维标准

## 4.2 设计原则

### P1. 对象优先，不以页面优先

先统一对象、关系、状态、生命周期，再统一流程和界面。

### P2. 三流协同，不以单流主导

统一设计 `控制流/任务流`、`数据流`、`工具流`，避免把平台退化为纯流程系统或纯数据系统。

### P3. 域内自治，平台共享

业务语义按域隔离，主数据、权限、文件、搜索、消息、规则、集成、审计等能力沉到平台共享层。

### P4. 模型驱动，代码兜底

通用变化通过对象配置、流程配置、规则配置、视图配置解决；高复杂度工程逻辑仍由专业服务与插件承接。

### P5. 数据可追溯，可审计，可回放

工业研发底座必须内生支持版本、基线、谱系、审批、变更、密级和归档。

### P6. 中心与边缘协同

试验和设备相关能力采用 `中心平台 + 边缘接入/缓存/同步` 模式，而不是全部强收口到中心。

### P7. 安全与国产化一等公民

保密、隔离、审计、国产化适配不能作为部署阶段附属工作，而应前置进入架构。

### P8. 复制优先，定制后置

优先沉淀行业模板包、初始化流程和标准化交付链路，再把差异化需求放到受控扩展层解决。

## 5. 业务域架构

## 5.1 一级业务域

建议按 `4+1` 业务域组织底座：

| 业务域 | 定位 | 代表产品/资产 | 核心对象 |
|---|---|---|---|
| 研发协同域 | 负责需求到任务到交付物的研发链协同 | `OLTran.CDM` | 需求、项目、WBS、工作包、任务、评审、变更、交付物 |
| 试验数据域 | 负责试验建模、采集、存储、分析、站点与虚实融合 | `OLTran.TDM` | 试验、测量、通道、设备、站点、数据集、测量文件 |
| 管理信息化域 | 负责科研管理、合同、设备器材、经营管理与主数据落地 | `科研业务管理系统`、`SUNBPM` | 计划、项目、合同、设备、借调、报废、组织、字典 |
| 平台共性域 | 负责共享基础能力与技术治理 | `OLTran` 共性能力 | 主数据、权限、文件、搜索、消息、规则、流程、连接器 |
| 知识与智能域 | 横切各域，负责知识沉淀与智能增强 | 知识中心、图谱、语义检索 | 知识资产、标准、经验、图谱实体、智能代理 |

## 5.2 端到端主业务链

### 链路 A：需求牵引的研发主线

`Requirement -> Project -> WBS/WorkPackage -> Task -> Design/Simulation/Test -> DataSet/Deliverable -> Review/Change -> KnowledgeAsset`

### 链路 B：试验驱动的数据主线

`Site/Device -> Acquisition -> Raw File/Series -> Standardized Model -> Data Center -> Analysis/Visualization -> Reuse by CDM/Knowledge`

### 链路 C：经营管理主线

`Annual Plan -> Research Project -> Contract/Fund -> Equipment Resource -> Approval/Statistics -> Management Dashboard`

### 链路 D：知识反哺主线

`业务对象/报告/模型/经验 -> 知识抽取 -> 图谱/索引 -> 检索/推荐/校核 -> 业务过程复用`

## 6. 统一对象模型

## 6.1 核心对象清单

建议底座优先固化以下核心对象：

| 对象 | 归属域 | 关键关系 |
|---|---|---|
| `Requirement` | 研发协同域 | 驱动项目、工作包、验证、知识 |
| `Project / Plan / WBS / WorkPackage` | 研发协同域、管理信息化域 | 连接需求、任务、资源、合同 |
| `Task / Workflow / Review / Change` | 研发协同域、平台共性域 | 连接对象状态、执行活动和审批链 |
| `DataSet / File / Deliverable / Model` | 研发协同域、试验数据域 | 连接设计、试验、分析、归档、知识 |
| `Test / Measurement / Channel / Device / Site` | 试验数据域 | 连接采集、分析、设备和站点 |
| `Tool / App / Template / Script` | 平台共性域 | 被任务调用，消费和产出业务数据 |
| `KnowledgeAsset` | 知识与智能域 | 关联对象、报告、经验和标准 |
| `Contract / Equipment / Payment / Borrow / Scrap` | 管理信息化域 | 与计划、项目、审批和资源关联 |
| `Organization / User / Role / Dictionary / MasterData` | 平台共性域 | 横切所有业务域 |

## 6.2 必须统一的对象治理规则

### 统一标识

- 所有核心对象必须具备全局唯一 ID。
- 外部系统主键、业务编码、展示编码分离管理。

### 统一生命周期

- 明确草稿、提交、审核、生效、归档、废止等状态。
- 对象状态变化必须进入可审计链路。

### 统一版本与基线

- 需求、模型、交付物、数据集、模板必须支持版本和基线。
- 基线应能与项目/WBS/里程碑关联。

### 统一密级与权限

- 密级、角色、组织、职责、业务对象权限联动治理。
- 权限模型建议采用 `RBAC + ABAC` 混合方式。

### 统一谱系

- 对象之间的来源、影响、上下游依赖必须可追溯。
- 谱系不只针对文件，也应覆盖需求、任务、模型、报告、试验结果。

建议采用七层能力结构：

1. **体验与入口层**
   - 统一门户
   - 角色工作台
   - 场景化导航
   - 看板与驾驶舱

2. **场景应用层**
   - 协同研发应用
   - 试验数据应用
   - 科研管理应用
   - 知识与智能应用

3. **领域服务层**
   - 需求服务
   - 项目/WBS/工作包服务
   - 任务/流程服务
   - 试验建模与数据服务
   - 合同/设备/计划服务

4. **平台共享服务层**
   - 主数据服务
   - 身份权限服务
   - 文件/对象服务
   - 搜索与标签服务
   - 消息通知服务
   - 规则服务
   - 流程引擎
   - 任务编排引擎
   - 模板服务

5. **数据与知识层**
   - 关系数据服务
   - 文档/对象数据服务
   - 图谱服务
   - 时序/测量数据服务
   - 搜索索引
   - 知识库与语义检索

6. **集成与边缘层**
   - API 网关
   - 事件总线
   - 文件交换
   - IT 连接器
   - OT 协议适配
   - 边缘缓存与同步代理

7. **治理与运行层**
   - 配置中心
   - 服务治理
   - 观测与告警
   - 安全与审计
   - 发布与灰度
   - 国产化适配

## 7.2 底座的七个核心能力域

| 能力域 | 核心能力 | 建设优先级 |
|---|---|---|
| 统一对象与元模型域 | 对象模型、扩展模型、状态、版本、基线、编码 | `P0` |

| 知识智能域 | 知识库、图谱、语义检索、推荐、LLM 辅助 | `P1` |
| 运行治理域 | 部署基线、观测、发布、安全、国产化 | `P0` |

## 8. 技术参考架构

## 8.1 推荐技术形态

整体建议采用：

- `领域服务 + 平台共享服务 + 集成中台 + 数据底座` 的分层结构
- 中心侧 `微服务/模块化单体混合演进`，避免一开始过度拆分
- 试验和设备接入侧采用 `边缘代理 + 消息/文件/缓存同步`
- 图谱、搜索、对象存储、关系库、时序库分工明确，不追求单库统管

## 8.2 部署拓扑建议

### 中心区

- 门户与应用集群
- 平台共享服务集群
- 领域服务集群
- 搜索、图谱、关系库、对象存储、消息与配置中心

### 集成区

- API 网关
- 外部系统连接器
- 工具执行代理
- 文件交换与同步服务

### 边缘区

- 试验站点接入网关
- 协议适配器
- 本地缓存与断点续传
- 批量数据预处理

## 8.3 存储治理矩阵

| 数据类型 | 建议主存储 | 说明 |
|---|---|---|
| 主数据、项目、合同、审批、权限 | 关系型数据库 | 强事务和结构化管理 |
| 扩展属性、柔性模型对象 | 文档数据库 | 支撑对象模型扩展 |
| 谱系、关系、主线、知识网络 | 图数据库 | 影响分析、知识图谱、对象关系探索 |
| 原始文件、模型、报告、多媒体 | 对象存储/文件存储 | 大文件与归档 |
| 测量/时序数据 | 时序库或列式库 | 采集数据、曲线数据 |
| 全文检索、标签检索 | 搜索引擎 | 搜索、导航、黄页和推荐召回 |

## 8.4 集成架构建议

底座统一暴露四类集成面：

1. **API 集成面**
   - 面向业务对象与流程调用
   - 强约束、可治理、可授权

2. **事件集成面**
   - 面向状态变化、异步同步、通知联动
   - 适合谱系、索引和消息异步更新

3. **文件集成面**
   - 面向大文件、批量数据、报告、模型交付
   - 需支持断点续传、校验、归档

4. **协议/代理集成面**
   - 面向 OT 协议、外部工具、本地脚本、HPC、设备侧数据链路
   - 通过代理和连接器进行隔离治理

## 8.5 CAE/SPDM/HPC 集成专题

针对 `SIM / VLab / OLink / 自主 CAE / HPC` 相关资产，建议在正式版中把仿真与算力协同提升为一级专题，至少包括：

- `Simulation Task Service`
- `HPC Job Proxy`
- `Result Package Service`
- `Warm/Cold Tier Policy`
- `Distributed File Mapping Strategy`

关键原则：

- 仿真任务编排与集群调度解耦
- 超大结果文件与业务元数据分离治理
- 结果包回流到 `Deliverable / KnowledgeAsset / TraceLink`
- 调度提交、回调、失败重试和人工接管可审计

## 9. 低/无代码能力定位与治理

## 9.1 结论

低/无代码能力需要补充，但不应作为底座主轴。建议将其定义为：

> 面向工业对象与受控业务场景的模型驱动装配能力，并通过统一元模型中心支撑柔性领域对象的受控派生与演化。

底座要防止两种偏差：

- 偏差 A：把低代码做成通用 OA/PaaS，最后难以承载工业语义
- 偏差 B：完全不做配置能力，导致每个项目都回到大量定制开发

## 9.2 应放在什么位置

低/无代码能力应位于：

- 业务域之上：承接场景化装配
- 平台共享服务之中：依赖统一对象模型、流程引擎、规则引擎、权限模型
- 治理体系之下：受发布、审计、模板、插件和环境管理约束

建议采用 `L1-L4` 四层扩展治理体系：

- `L1 内核层`：核心对象语义、权限/密级、TraceLink、租户隔离，只能升级
- `L2 平台能力扩展层`：连接器、规则脚本、AI Worker、SDK 服务，由平台开发和发布治理控制
- `L3 领域模型扩展层`：白名单对象派生、SchemaVersion、行业模板包，由实施架构师和模型审批流控制
- `L4 场景装配层`：页面、流程、看板、查询配置，由实施顾问或客户管理员受控操作

它不应直接绕过：

- 核心对象模型
- 权限与密级规则
- 版本与基线治理
- 审计与发布流程

## 9.3 元模型驱动扩展体系

低/无代码不应只停留在页面装配层，还应补入 `元模型驱动扩展体系`，重点支撑：

- 柔性领域对象派生
- 模型版本与审批
- 模型发布与回滚
- 模型变更审计
- TDM 侧的 `TestSpec / MeasurementSchema` 动态演化
- 模型发布后的权限点、审计类别、索引和基础视图自动联动

边界要求：

- 核心共享对象不允许被前台重定义
- 白名单柔性对象允许受控派生
- 存储映射应支持文档库、扩展字段和图谱实体协同演化
- 模型变更必须进入统一发布链路

## 9.3.1 行业模板包（ITP）

为避免“有配置能力但没有复制载体”，建议正式引入 `Industry Template Package (ITP)`：

- 封装行业模型、流程、视图、规则、主数据初始化和连接器配置
- 绑定 `SchemaVersion`、环境变量、依赖关系和插件挂载点
- 通过统一发布、回滚和差量升级链路生效
- 作为 `3-4 周快速复制交付` 的标准载体

## 9.4 适用场景

建议优先覆盖以下场景：

- 门户、工作台、导航页、业务看板
- 表单、台账、查询页、配置页
- 审批流程、任务流程、通知流程
- 轻量规则配置
- 轻量集成编排
- 报表和统计分析
- 面向不同单位/型号/项目的差异化装配

## 9.5 不宜由低/无代码承接的场景

- 复杂研发协同语义建模
- 深度 CAD/CAE/PDM/TDM 工具集成
- 大规模测量数据处理与高性能分析
- 强保密、强审计、强一致性的核心链路
- 高复杂度工程算法、优化计算和 HPC 调度

## 9.6 能力清单

建议把低/无代码拆成九类能力：

| 能力 | 说明 | 优先级 |
|---|---|---|
| 对象配置 | 扩展对象字段、状态、校验、展示模型 | `P1` |
| 元模型派生 | 白名单柔性对象的受控派生与模型演化 | `P0` |
| 行业模板包/初始化 | ITP、租户初始化、模板变量注入、差量升级 | `P0` |
| 表单与视图配置 | 列表、详情、表单、树、卡片、看板 | `P1` |
| 流程配置 | 审批流、任务流、通知流、回调编排 | `P1` |
| 规则配置 | 校验、计算、触发、策略判断 | `P1` |
| 门户与看板配置 | 工作台、角色门户、驾驶舱 | `P1` |
| 集成编排配置 | API 调用、文件交换、事件编排 | `P1` |
| 插件与组件扩展 | 专业组件、领域控件、工具适配器 | `P2` |

## 9.7 治理原则

### 受控模型驱动

- 所有低代码对象必须挂靠到底座统一元模型。
- 不允许在低代码层随意创造脱离治理的“新主数据体系”。
- 柔性领域对象的派生必须通过对象注册中心和模型审批链路。

### 模板优先

- 优先提供行业模板、单位模板、场景模板，而不是开放式白板搭建。
- 模板包的 `manifest.yaml` 必须预留插件挂载点和依赖声明。

### 插件化扩展

- 对复杂专业逻辑，采用插件/脚本/服务扩展，而不是在可视化配置器中强行表达。

### 发布隔离

- 配置变更需走环境隔离、发布审批、版本回滚和审计记录。

### 资产化运营

- 表单模板、流程模板、看板模板、连接器模板要进入模板仓和资产目录。

### 安全兜底

- 所有低代码运行时仍统一经过权限、密级、日志和审计链路。

## 9.8 建设顺序

### Phase 1（现在）

- ITP
- MetaObjectType / SchemaVersion
- 租户初始化
- 表单/视图/流程基础装配
- 白名单柔性对象派生

### Phase 2（12 个月后）

- 发布后自动联动注册
- 受控的模型驱动 API / 基础视图生成
- 更完整的差量升级和模型合并策略

### Phase 3（24 个月后）

- 插件 SDK
- 连接器市场
- 行业模板市场
- 受控 AI 辅助搭建

## 10. 治理体系设计

## 10.1 对象治理

- 建立统一对象词典和对象责任矩阵
- 明确对象归属域、主责团队、共享范围和变更流程
- 所有核心对象必须沉淀 ADR 级定义

## 10.2 数据治理

- 统一数据分级分域分责
- 统一标签、检索、归档、脱敏、审计规则
- 为文件、对象、时序、图谱分别定义治理标准

## 10.3 API 与集成治理

- 建立连接器目录与接口目录
- 统一认证授权、配额、审计、版本策略
- 明确同步、异步、文件、协议四类集成标准

## 10.4 版本与发布治理

- 业务对象版本、模板版本、服务版本、接口版本分别治理
- 低代码配置、规则和模板必须纳入发布流程
- 关键对象支持基线冻结与回溯
- `SchemaVersion`、`ITP` 和租户初始化计划必须进入统一发布链路

## 10.5 安全与保密治理

- 采用 `租户/组织/项目/对象/字段` 多层权限模型
- 密级、审批、审计、导出、打印、外发链路统一管控
- 对边缘接入和跨网传输单列安全策略

## 10.6 平台与产品治理

- 平台团队负责共享能力和技术基线
- 业务团队负责领域服务和业务语义
- 产品线治理委员会负责对象、接口和版本仲裁

## 10.7 租户与部署隔离治理

正式版需要补入 `租户与部署隔离策略矩阵`，至少覆盖三类模式：

1. 单法人单实例
2. 多法人逻辑隔离
3. 高保密客户物理隔离部署

每种模式都应定义：

- 服务隔离方式
- 数据隔离方式
- 对象存储隔离方式
- 搜索与消息隔离方式
- 主数据订阅与分发策略

## 10.8 AI Worker 治理

建议将 AI 从单纯“侧挂问答”升级为 `受控 AI Worker` 路线：

- 允许 AI 作为流程中的自动化执行节点
- 先应用于预审、校验、摘要、分类、推荐
- 不允许直接替代最终责任人和最终提交者
- 必须有提示词版本、知识来源、审计和回放

## 10.9 复制交付治理

正式版需要把复制交付从“期望”变成“治理对象”：

- 定义模板包版本、兼容性和升级责任
- 定义初始化流程的审批、执行和回滚责任
- 建立交付周期、复用率、人月节省和 ROI 指标
- 用 `Phase 0` 纵切先验证 3-4 周上线目标是否成立

## 12. 需要尽快确认的架构决策

建议下一轮直接围绕以下决策形成正式 ADR：

1. 底座最终是 `单一产品底座` 还是 `产品族共用平台内核`。
2. `CDM/TDM/MIS` 是否共享同一身份、权限、主数据和文件服务。
3. 试验数据的中心与边缘边界如何划分。
4. 图数据库、时序库、对象存储是否进入标准底座选型。
5. 低/无代码与元模型扩展允许作用于哪些柔性领域对象。
6. `SUNBPM` 在未来定位为流程引擎能力，还是完整管理应用产品族。
7. 平台要支持哪几种租户与部署隔离模式，以及各自的主数据分发边界。
8. CAE/SPDM/HPC 场景是否上升为标准底座能力，并采用何种代理与结果包策略。
9. AI Worker 允许进入哪些流程节点，以及哪些节点必须人工兜底。
10. `ITP` 的格式、加载顺序和插件挂载点如何冻结。
11. 发布后自动联动做到什么边界，何时进入 API 自动生成。
12. 核心对象与柔性对象采用何种混合存储策略。

## 13. 本草案对下一步工作的直接指导

如果下一步进入“底座架构设计与治理”正式版，建议按以下顺序展开：

   这是连接现有系统和防止烟囱再生的关键。

4. 最后出《产品装配与低/无代码治理方案》  
   让差异化交付建立在受控平台能力之上，而不是反过来侵蚀底座。

## 15. 当前版本结论

现阶段最重要的不是继续堆产品清单，而是尽快锁定三件事：

- 底座要统一的 `对象`
- 底座要沉淀的 `共享能力`
- 低/无代码要遵守的 `边界和治理`

如果这三件事不先定清楚，后续任何技术架构、部署架构和治理机制都会重新回到项目化定制。
