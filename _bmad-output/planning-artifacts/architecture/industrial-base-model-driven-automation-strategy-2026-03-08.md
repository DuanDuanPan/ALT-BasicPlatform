# 工业研发底座模型驱动自动联动策略

状态：`Draft v0.1`  
日期：`2026-03-08`  
用途：定义 `MetaObjectType / SchemaVersion` 发布后，平台如何自动联动注册，以及 Phase 2 如何演进到模型驱动 API 自动生成。

---

## 1. 目标

这份策略解决的问题不是“能不能生成 CRUD 接口”，而是：

1. 模型注册后哪些平台能力必须自动同步
2. 自动联动到什么程度才不破坏治理
3. 什么时候进入 API 自动生成

---

## 2. 总原则

推荐采用两阶段策略：

### Phase 1

`发布后自动注册`

### Phase 2

`模型驱动 API 自动生成`

不建议直接一步跳到“注册即生效 + API 全自动生成”。

---

## 3. Phase 1 自动联动范围

当 `MetaObjectType / SchemaVersion` 发布后，建议自动触发以下联动：

- 搜索索引注册
- 权限点注册
- 审计点注册
- 基础事件主题注册
- 基础视图元数据注册
- TraceLink 关系类型注册

这意味着：

- 新派生对象发布后，不需要手工到五个系统分别登记
- 但仍然要经过发布闸门

---

## 4. Phase 2 自动生成范围

当 Phase 1 稳定后，再进入 Phase 2：

- 标准 CRUD API 生成
- 标准 Query API 生成
- 标准事件定义生成
- 标准权限点清单生成
- 标准管理页骨架生成

限制：

- 只针对白名单柔性对象
- 不对核心对象直接自动生成替代领域服务的接口

---

## 5. 自动联动管线

建议引入统一发布管线：

```text
对象注册 -> 模型校验 -> 审批 -> 发布
      -> 自动联动编排器
         -> Search Registry
         -> Permission Registry
         -> Audit Registry
         -> Event Registry
         -> View Metadata Registry
         -> Trace Type Registry
```

---

## 6. 自动联动边界

自动联动允许：

- 注册元数据
- 注册权限点
- 注册索引结构
- 注册视图骨架

自动联动不允许：

- 自动越权发布
- 自动修改核心对象语义
- 自动绕过人工审批
- 自动跨租户传播

---

## 7. 与运行时加载的关系

Party Mode 讨论提出的“运行时动态模型加载”方向是对的，但应采用保守落地策略：

### 当前阶段

- 发布后运行时可识别
- 注册后可预编排
- 已发布模型可被查询、索引、授权和审计

### 后续阶段

- 运行时自动生成标准访问层
- 运行时自动挂载标准管理接口

---

## 8. 与存储路由的关系

自动联动必须读取 `MetaObjectType` 的存储路由信息：

- 核心对象 -> 固定主结构
- 柔性对象 -> JSON/文档扩展
- 图谱对象 -> 关系类型注册

没有存储路由，自动生成的 API 会缺少正确的数据落点。

---

## 9. 验收标准

Phase 1 的验收标准建议为：

- 新发布一个白名单对象派生后
- 平台能自动完成索引、权限、审计、视图元数据的注册
- 无需手工修改多个子系统配置
- 全过程可审计、可回滚

Phase 2 的验收标准建议为：

- 能生成标准 CRUD/Query API
- 能生成标准事件定义
- 能为管理端生成基础页面骨架

---

## 10. 当前结论

模型驱动自动化不应被理解成一个“炫技功能”。  
它的真实意义是：

- 降低新增对象的集成成本
- 降低模板升级成本
- 降低交付配置的人力成本

所以 Phase 1 先做“自动联动”，Phase 2 再做“自动生成”，这个顺序是合理的。
