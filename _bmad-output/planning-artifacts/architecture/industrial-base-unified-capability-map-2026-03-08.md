# 工业研发底座统一能力地图

状态：`Draft v0.3`  
日期：`2026-03-08`  
用途：作为《工业研发底座架构设计与治理草案》的配套产物，用于明确 `CDM/TDM/MIS/平台共性/知识智能` 的能力边界、共享程度、建设优先级和治理归属。

---

## 1. 能力地图目标

这份能力地图解决五件事：

1. 明确哪些能力必须沉到 `底座共享层`。
2. 明确哪些能力仍应保留在 `领域专属层`。
3. 明确哪些能力允许由 `低/无代码装配层` 承接，哪些绝不允许。
4. 为后续 `统一对象模型`、`ADR`、`产品边界` 和 `建设路线图` 提供统一语言。
5. 把“快速复制交付”拆成清晰的平台能力，而不是停留在战略口号层。

---

## 2. 能力地图总览

## 2.1 五类能力域

| 能力域 | 说明 | 主要承载对象 |
|---|---|---|
| 研发协同能力域 | 面向需求、项目、工作包、任务、评审、交付物的协同执行能力 | `CDM` |
| 试验数据能力域 | 面向试验建模、采集控制、数据处理、分析与站点接入能力 | `TDM` |
| 管理信息化能力域 | 面向计划、项目、合同、设备与经营管理能力 | `MIS/SUNBPM` |
| 平台共性能力域 | 面向对象模型、主数据、文件、流程、权限、集成、运行治理能力 | `OLTran` 共性底座 |
| 知识与智能能力域 | 面向知识沉淀、图谱、检索、推荐、AI 辅助能力 | 横切各产品线 |

## 2.2 三层分工原则

### 领域专属层

- 承载强业务语义和领域规则
- 由业务产品团队主导
- 不应被通用低代码能力替代

### 平台共享层

- 承载多域复用的核心底座能力
- 由平台团队统一治理
- 必须对外提供标准服务和标准规则

### 装配配置层

- 承载门户、表单、流程、规则、看板、轻量集成等可配置能力
- 必须依赖统一对象模型和权限体系
- 不能绕开底座治理

## 2.3 四层扩展治理视图

为避免“配置能力失控侵入内核”，建议把扩展治理叠加为四层：

| 层级 | 说明 | 主要操作者 |
|---|---|---|
| `L1` 内核层 | 核心对象语义、权限/密级、TraceLink、租户隔离 | 平台核心团队 |
| `L2` 平台能力扩展层 | 连接器、规则脚本、AI Worker、SDK 服务 | 平台开发者 |
| `L3` 领域模型扩展层 | 白名单对象派生、SchemaVersion、行业模板包 | 实施架构师 |
| `L4` 场景装配层 | 页面、流程、看板、查询装配 | 实施顾问/客户管理员 |

## 4.2 低/无代码可承接性矩阵

| 能力 | 是否允许低代码承接 | 说明 |
|---|---|---|
| 门户/导航/工作台 | 是 | 适合差异化装配 |
| 表单/查询/看板 | 是 | 适合快速交付 |
| 审批流/通知流 | 是 | 适合配置化 |
| 轻量规则 | 是 | 需受规则引擎约束 |
| 轻量集成编排 | 是 | 需经过连接器与发布治理 |
| 柔性领域对象派生 | 有条件 | 仅限白名单对象，并纳入模型版本、审批与回滚 |
| 核心对象定义 | 否 | 必须由统一元模型治理 |
| 核心权限模型 | 否 | 不允许绕开安全治理 |
| 深度 CAD/CAE/PDM/TDM 集成 | 否 | 属于专业工程能力 |
| 大规模测量数据处理 | 否 | 属于专用数据能力 |
| 保密主链路 | 否 | 必须保留强治理实现 |

## 5. 能力归属与组织分工

## 5.1 归属原则

### 平台团队负责

- 统一对象模型
- 元模型扩展体系
- 行业模板包运行时与模板资产目录
- 主数据
- 身份权限
- 文件与对象服务
- 流程/任务引擎
- 搜索、审计、配置、发布、观测
- 集成网关、事件总线、连接器框架
- 低/无代码装配运行时和模板体系
- 模型发布自动联动运行时
- Worker 执行运行时
- 租户与部署隔离基线

### 业务产品团队负责

- `CDM`：需求、项目、WBS、工作包、协同执行、评审变更
- `TDM`：试验建模、采集控制、站点接入、测量数据处理、分析应用
- `MIS`：科研计划、合同、设备、经营分析与台账

### 联合治理负责

- 知识图谱
- 核心对象词典
- 共享模板资产
- 连接器目录
- 低/无代码边界和禁区

## 5.2 建议建立的治理机制

| 机制 | 目标 | 参与方 |
|---|---|---|
| 架构治理委员会 | 仲裁共享与专属边界、选型和标准 | CTO、架构、平台负责人、产品负责人 |
| 对象治理小组 | 统一对象词典、状态、版本、基线规则 | 架构、产品、数据治理 |
| 集成治理小组 | 统一接口标准、连接器目录、外部系统接入规范 | 平台、集成、运维、安全 |
| 低代码治理小组 | 管控模板、插件、发布、审计与边界 | 平台、产品、交付、安全 |
| 模板包治理小组 | 管控 ITP 标准、兼容性、初始化流程和升级策略 | 平台、产品、交付、架构 |

## 7. 对下一步对象模型的直接输入

能力地图已经指向了对象模型设计的重点：

1. 优先定义 `Requirement / Project / WorkPackage / Task / DataSet / Test / Device / Contract / Organization / KnowledgeAsset`。
2. 对 `流程、任务、数据集、文件、模板` 这些半共享能力，要拆成 `共享内核 + 域内增强`。
3. 对 `低/无代码`，只允许配置和装配，不允许创造独立主数据和独立安全模型。
4. 对 `ITP / SchemaVersion / 模型自动联动`，必须作为复制引擎核心能力进入对象模型和 ADR。

## 9. 当前版本结论

统一能力地图的核心价值，不是把所有产品能力都抹平，而是把它们拆成三类：

- 必须共享的平台能力
- 底座内核加领域增强的半共享能力
- 必须保留在领域内部的专属能力
