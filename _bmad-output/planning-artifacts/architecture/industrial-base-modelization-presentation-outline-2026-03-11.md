## Slide 1：执行摘要——本次架构专题的核心结论与决策点

- **页面目标：** 让评审参与者在第一页就知道核心结论、架构价值和需要对齐的决策点
- **视觉建议：** 四宫格，分别对应“为什么现在做 / 已形成什么共识 / 预期价值 / 需要拍什么板”
- **页面核心内容：**
  - **为什么现在必须做：**
    - 现有底座具备配置工具，但缺少可沉淀、可安装、可治理的业务资产载体
    - 项目交付仍然高度依赖公司侧按项目单独建模开发，现场再导入、调整、验证，导致复用弱、演进难、升级风险高
  - **这次已形成的核心共识：**
    - 以 `ITP` 作为模型化资产载体
    - 采用 `M0 / M0.5 / M1 / M2` 模型继承轴
    - 实例数据不进入 `M` 轴，运行时绑定 `schemaVersionRef`
    - 运行时能力收敛为 `Object Model / Task Orchestration / Workflow / Toolflow`，由 `Trace Service` 提供横切协同
    - 完整六层架构体系（P0 产品基线 → P5 引擎与组件），形成层次清晰的产品架构框架
    - UI 架构采用"引擎 + Schema"混合模式（AD-24），支持 NocoBase 式原地编辑
    - FED（Feature Extension Descriptor）机制实现编辑态自动产出、便携性分级和跨模块协调
    - 平台→产品→项目逐级支撑，项目→产品→平台逐级积累的双向流动闭环
  - **预期价值：**
    - 交付方式从“公司侧按项目建模开发 + 现场导入调整”转向“标准 ITP 导入 + 受控差量”
    - 领域专有功能从“沉在单项目方案里”转向“沉淀为可复用的产品资产和能力模块”
    - 在统一底座上通过 ITP 组合和差量扩展，更快覆盖相邻行业场景
    - 客户差异化从“改坏标准包”转向“标准包之上的受控差量”
    - 模型、依赖、主线和影响分析进入统一治理闭环
  - **本次希望拍板事项（4 项）：**
    - 认可”模型化资产交付”作为底座核心演进方向
    - 认可 ITP + M 轴 + 运行时职责边界的总体方案
    - 同意以 TDM 风洞试验场景作为首个标杆域
    - 同意按”双线推进”方式组织实施：底座修路 + 标杆域造车
- **建议口播：**
  > 今天这场架构专题评审，核心目标是对齐底座演进方向。核心结论是：模型化不是给现有建模器换个名字，而是把业务概念上升为一等架构资产，并通过 ITP、M 轴和运行时治理体系，把交付方式从”工具交付”改造成”资产交付”。以下各页会逐一展开技术细节，供各位专家评审。

## Slide 2：问题不在工具不够，而在资产没有沉淀

- **页面目标：** 把原来三页痛点压缩成一页，避免铺垫过长
- **视觉建议：** 左侧“旧模式工作流”，右侧“结果问题”，中间用红色箭头标出断裂点
- **页面核心内容：**
  - **现状：**
    - 有表单、流程、数据建模工具
    - 也支持导入导出、规则配置和二次扩展
  - **真正的问题：**
    - 输出的是零散配置，不是可安装的业务资产
    - 公司侧仍需按项目单独建模开发、跨模块拼装、写胶水逻辑，现场再继续导入、调整和验证
    - 导出的多是 SQL、XML、碎片配置，缺少业务上下文、依赖关系和治理规则
  - **直接后果：**
    - 复用困难：换一个客户，核心工作大多重来
    - 资产流失：标杆项目经验难以沉淀为标准包
    - 产品化不足：领域专有功能沉在项目方案里，难以变成标准产品能力
    - 行业扩展慢：进入相邻行业或新场景时，仍需从单项目方案重新改造
    - 升级高风险：平台升级容易牵动客户个性化配置
    - 治理薄弱：缺少影响分析、依赖图谱和可追溯主线
- **一句话结论：**
  - **我们不是缺建模工具，而是缺“业务资产的正式载体”和“围绕资产运行的治理机制”。**
- **建议口播：**
  > 过去我们一直在增强工具能力，但真正的断点不在工具本身，而在于这些工具产出的内容并没有被收敛成一个完整、可复用、可治理的业务资产。因此每个项目看似都在复用配置，实质上仍然是公司侧为单个项目生产一次性方案，再到现场做导入、适配和验证。

## Slide 3：什么是模型化——把业务概念提升为一等架构资产

- **页面目标：** 定义模型化，并引出 `ITP`
- **视觉建议：** 左侧“碎片能力”，右侧“ITP 资产包”，中间用“升维”箭头连接
- **页面核心内容：**
  - **核心定义：**
    - **模型化 = 把业务概念从硬编码和零散配置，提升为平台可注册、可版本化、可自动联动、可差异化扩展、可标准化交换的一等架构资产。**
  - **模型化的资产载体：`ITP`（Industry Template Package）**
    - `Schema`：业务对象、关系、数据组织方式
    - `Templates`：活动模板、生命周期、管控流程、事件联动
    - `Behavior`：视图、页面、扩展挂接
    - `Runtime`：主数据要求、连接器契约、合规规则、种子数据
  - **它带来的交付方式变化：**
    - 从“公司侧按项目建模开发 + 现场导入调整”转向“底座 + 标准 ITP 导入 + 受控微调”
    - 从“项目代码资产”转向“领域业务资产”
    - 从“领域功能沉在项目里”转向“领域功能沉淀为产品能力，并通过 ITP 快速扩展行业应用”
- **建议口播：**
  > 模型化不是把旧有低代码体系重新命名，而是把原本散落在表结构、流程定义、脚本和页面配置里的业务知识，重新组织为可注册、可装载、可治理的正式资产。这份资产的载体，就是 ITP。更重要的是，过去沉在单项目里的领域专有功能，可以开始沉淀为标准产品能力，再通过 ITP 组合和差量扩展，更快覆盖新的行业场景。

## Slide 3.5：完整层次结构——从平台到项目的六层架构

- **页面目标：** 一张图让所有听众建立统一的架构心智模型，理解 ITP、M 轴、编辑态在架构中的位置
- **视觉建议：** 六层横向堆叠图（P5 底→P0 顶），右侧叠加 M 轴色带（M0 蓝 / M0.5 青 / M1 绿 / M2 橙），以工作包+绩效扩展为场景标注
- **页面标题建议：** 从平台到项目的六层架构——每一层做什么、谁负责
- **页面核心内容：**

```text
┌──────────────────────────────────────────────────────────────┐
│ P0  产品基线    OLTran.CDM v1.0 → 航天院 CDM 定制版          │
├──────────────────────────────────────────────────────────────┤
│ P1  能力包      project-execution-pack（交付单元）            │
│                 + 包级编排模块 + FED 协调                      │
├──────────────────────────────────────────────────────────────┤
│ P2  模块        work-package-center（定义单元）               │
├──────────────────────────────────────────────────────────────┤
│ P3  资产        PageTemplate(UI Schema) / WorkPackage 对象    │
│                 ActionModel / FlowModel / FeatureExtension    │
├──────────────────────────────────────────────────────────────┤
│ P4  构件        ExtensionPoint / FieldBinding / Draft          │
│                 （编辑态在这里操作）                            │
├──────────────────────────────────────────────────────────────┤
│ P5  引擎+组件   graph-canvas / table-engine / io-configurator │
│                 ServerUnit / ClientUnit（代码级扩展单元）       │
│                 （硬编码引擎 + Schema 灵活 + Extension Unit 逃逸舱）│
└──────────────────────────────────────────────────────────────┘
         M0 平台内核    M0.5 公共业务    M1 领域 ITP    M2 客户差异
```

- **这页要讲透的 4 个点：**
  - 每一层有明确的职责边界——P5 是能力地板，P2 是定义者，P1 是装配者，P0 是出厂形态
  - ITP 不是新概念，它就是 P1-P3 的打包产物
  - M2 编辑态操作 P3-P4 层（零代码），超出零代码边界时通过 P5 的 Extension Unit（ServerUnit/ClientUnit）代码级扩展——但仍在 FED 统一管理体系内
  - 右侧 M 轴色带表示"谁控制"——蓝色平台团队年级更新，橙色客户月级扩展
  - 用户在一个页面上操作时，P0-P5 多层同时生效，不是逐层深入——一个工作包分解页面背后有 5 个 P 层在协同工作
- **建议口播：**
  > 这张图是理解整个底座架构的"地图"。从底往上看：P5 是平台的交互引擎，硬编码保证品质，比如图形化分解树的拖拽引擎。P3 是模块的内容物——页面模板、业务对象、流程规则和特性扩展。P1 是能力包，面向客户的交付单元。ITP 就是把 P1 到 P3 打成一个可安装的包。右侧的颜色代表谁控制——平台团队管蓝色和青色层，领域团队管绿色层，客户和实施管橙色层。即使需要写代码的复杂场景——比如自定义分析算法、调用商业软件——也通过 Extension Unit 注册到 P5 层，由 FED 统一管理，不会变成脱管的项目私有代码。客户在编辑态做的每一个扩展，都在 P3-P4 的受控范围内。

## Slide 6：关键架构决策一——`M` 轴分层与实例数据边界

## Slide 6.5：双向流动——项目积累如何反哺产品和平台

- **页面目标：** 讲透"飞轮效应"——每一个客户项目的 M2 定制，都在为平台未来能力做增量积累
- **视觉建议：** 循环流动图（平台 → 产品 → 项目 → 产品 → 平台），中心标注"飞轮效应"
- **页面标题建议：** 越用越强——项目积累如何反哺产品和平台
- **页面核心内容：**

```text
         ┌──── 平台能力扩大 ←───── 跨领域共性提炼 ─────┐
         │                                              │
         ▼                                              │
   ┌──────────┐                                  ┌──────────┐
   │  平台     │ ── 引擎+组件+扩展点 ──→          │  产品     │
   │  M0/M0.5 │                                  │  M1 ITP  │
   └──────────┘                                  └──────────┘
         │                                              │
         │ 支撑                                   交付  │
         │                                              │
         │         ┌──────────┐                         │
         └────→    │  项目     │  ←──── 能力包 ─────────┘
                   │  M2 实施  │
                   └──────────┘
                         │
                   编辑态 → FED
                         │
                   多项目 FED → M1 提炼标准
                   多领域模式 → M0 抽象通用
```

  - **自顶向下（逐级支撑）：**
    - 平台提供引擎 + 扩展点机制 → 产品团队用它构建标准能力包 → 客户项目在此基础上扩展
    - 每一层的承诺是下一层安全操作的前提
  - **自底向上（逐级积累）：**
    - 客户编辑态产出 FED → 多个项目的相似 FED → 领域团队提炼为标准能力包 → 纳入 ITP
    - 多个领域出现相似模式 → 平台团队抽象为通用引擎或组件 → 平台能力边界扩大
    - **代码级扩展同样积累：** 客户项目的 ServerUnit（如振动分析算法）→ 领域团队标准化接口 → 平台团队抽象为通用能力。今天写代码解决的问题，明天可能变成平台的零代码标准能力。
  - **晋升条件（不是所有 FED 都应该晋升）：**
    - M2→M1：≥2 项目验证 + 无客户专有逻辑 + 领域评审
    - M1→M0：≥2 领域复现 + 通用性抽象 + 平台接管维护
  - **一句话核心：**
    - **项目做得越多 → FED 积累越丰富 → 领域 ITP 越完整 → 新客户交付越快 → 更多项目 → 更多积累**
- **建议口播：**
  > 这是整个模型化架构最重要的商业逻辑。从上往下看，平台提供引擎和扩展点，领域团队构建标准产品能力，客户在此基础上做差异化扩展。从下往上看，客户的每一次定制都自动生成结构化的特性扩展记录。当多个项目做了相似的扩展，领域团队就可以提炼为标准能力包纳入 ITP。当多个领域出现相似模式，平台团队就可以抽象为通用组件。这是一个越转越快的飞轮——项目做得越多，产品越完整，下一个项目交付越快。而且这个飞轮不仅适用于零代码扩展——代码级的扩展单元也走同一条积累路径。一个客户的自定义分析算法，经过多项目验证后可以成为标准产品能力，甚至沉淀为平台内置引擎。这才是"从工具交付走向资产交付"的真正价值所在。

## Slide 7：关键架构决策二——运行时服务边界与状态归属

- **页面目标：** 修正“谁持有业务真相状态”的常见误解
- **视觉建议：** 四服务协同图，突出“谁管什么状态、谁不管什么状态”
- **页面核心内容：**
  - **`L3` 领域聚合：**
    - 持有业务对象真相状态
    - 如 `Requirement / WorkPackage / Test / Contract`
  - **`Task Orchestration`：**
    - 持有活动实例状态、生命周期阶段状态、事件联动执行状态
    - 解释并执行活动模板、阶段模板和事件链声明
  - **`Workflow Service / Activiti`：**
    - 只持有人工审批子流程状态
  - **`Toolflow Orchestration`：**
    - 只持有工具执行运行时状态
  - **`Trace Service`：**
    - 作为横切能力，承接 `TraceIntent`，生成 `TraceLink`
  - **一句话结论：**
    - **业务真相不交给编排层托管，编排层只管理编排实例状态。**
- **建议口播：**
  > 这页是整场汇报里最容易被技术专家追问的地方。我们这次不是再造一个万能状态机，而是把状态边界理顺。L3 领域聚合持有业务真相，Task 编排层只管理编排实例，Workflow 只做审批，Toolflow 只做工具执行，Trace 负责横切追踪。这种收敛方式，才真正有利于模型化落地后的稳定运行。

## Slide 8：关键架构决策三——治理闭环：依赖图谱、影响分析与 TraceLink

- **页面目标：** 把“高级能力”讲成“模型化成立后的自然治理结果”
- **视觉建议：** 左边模型依赖图谱，右边实例主线图谱，中间用统一 Graph Service 串联
- **页面核心内容：**
  - **模型级治理：依赖图谱 + 影响分析**
    - `ITP` 装载时自动扫描 `schema / templates / behavior / runtime / semantic-refs`
    - 构建模型级依赖图谱
    - 删除、修改、裁剪前给出直接影响、间接影响、阻断或降级建议
  - **实例级治理：TraceLink 主线**
    - 模型定义关系类型
    - 运行时对象产生 `TraceIntent`
    - `Trace Service` 幂等生成或更新 `TraceLink`
    - 主线服务基于 `TraceLink` 计算正向追踪、反向溯源和变更影响
  - **统一观点：**
    - 模型级依赖图谱和实例级主线图谱，本质上都是“图上的关系管理和遍历”
    - 可由同一个图服务体系支撑
- **建议口播：**
  > 影响分析和 TraceLink 不是后加的亮点功能，而是模型化以后必然出现的治理闭环。前者解决“改之前是否知道会影响什么”，后者解决“运行之后能否追踪、溯源和分析影响”。一个面向模型定义，一个面向运行实例，本质上都是图关系管理问题。

## Slide 9.5：首个标杆域比选——为什么建议从 TDM 切入

- **页面目标：** 回答"为什么选 TDM 而不是 CDM 或 MIS 先打样"，主动封堵现场质疑
- **视觉建议：** 三列对比 + 下方总结栏
- **页面核心内容：**
  - **三域定性对比：**

    | 对比维度 | TDM | CDM | MIS |
    |---|---|---|---|
    | 对象模型丰富度 | 最丰富——schema 填充最重，最能验证 ITP 承载力 | 中等 | 较简单 |
    | 编排与集成依赖 | 最轻——不依赖尚未收敛的编排能力 | 最重——连接器密集，编排复杂 | 较重——规则编排密集 |
    | 数据主线完整度 | 最完整——天然具备需求→试验→数据→报告全链路 | 部分——需跨域才能闭环 | 弱——管理对象主线较短 |
    | 已有积累 | 风洞场景已有标杆项目，可直接提炼标准 ITP | 有积累但分散 | 有积累但分散 |
    | 试点风险 | 低——不依赖底座未就绪能力 | 高——编排和连接器能力需先就绪 | 高——差量和合规机制需先闭环 |

  - **关键判断逻辑：**
    - **TDM 的核心优势：** 对象模型最丰富（schema 填充最重），天然具备完整数据主线，可以一次性验证 ITP 包结构、M 轴继承、Schema 注册、TraceLink 自动生成和影响分析等核心机制
    - **CDM 为什么不宜首选：** 编排逻辑和连接器依赖最重，而 Task Orchestration 的活动模板 schema 和详细时序仍在待细化清单中，首轮贸然切入容易卡在底座能力未就绪
    - **MIS 为什么不宜首选：** 规则极度个性化，M2 差量和合规约束机制是其核心价值，但合规声明 schema 和兼容性引擎尚未定稿，先打容易暴露过多未收敛问题
    - **后续扩展路径：** TDM 验证闭环后 - CDM 扩展编排和连接器能力 - MIS 扩展规则和合规能力
  - **一句话结论：**
    - **TDM 是"模型最重、编排最轻、主线最完整"的域，最适合作为首个标杆域验证 ITP 核心架构机制，同时具备从已有项目中提炼标准 ITP 的现实条件。**
- **建议口播：**
  > 为什么建议率先在 TDM 切入？核心原因有三个。第一，TDM 的对象模型最丰富，最能检验 ITP 包结构和 Schema 注册能力。第二，TDM 天然具备从需求到报告的完整数据主线，可以一次性验证 TraceLink 和影响分析。第三，TDM 对编排和外部集成的依赖最轻，不会因为底座尚未收敛的编排能力而卡住试点进度。CDM 和 MIS 则分别在编排和规则维度更重，更适合在底座侧相应能力就绪后再逐步扩展。

### Appendix F：成功度量指标框架

- **用途：** 为模型化架构落地提供方向性验证基线，供标杆域试点阶段和后续推广阶段对照使用
- **度量设计原则：** 首轮试点以方向性区间为目标，不预设绝对数字；待首个 ITP 闭环后，根据实测数据校准目标值

| 指标 | 定义 | 旧模式基线（估） | 首个 ITP 试点方向 | 度量方式 |
|---|---|---|---|---|
| **交付周期** | 从需求确认到客户可运行环境交付的日历时间 | 月级 | 目标缩短至周级 | 项目里程碑记录 |
| **复用率** | 标准 ITP 内的 Schema / Template / Behavior 被后续项目直接复用（含 M2 差量叠加）的比例 | 低（配置碎片手工搬运） | 显著提升（首批同域项目验证后校准具体数字） | ITP 安装日志 + 差量统计 |
| **升级成功率** | 底座或 ITP 版本升级后，客户环境一次性通过兼容性校验并正常运行的比例 | 低（升级常需现场逐项排查） | 显著提升（基于 SchemaVersion 兼容性引擎 + 预检报告） | 升级预检报告 + 回归验证结果 |
| **客户差量占比** | M2 差量（客户个性化）占整个 ITP 总资产的比例 | 无法度量（标准与个性化混杂） | 标准包占比应显著高于差量占比（试点后校准具体比例） | ITP manifest 中 M2 overrides 统计 |

- **辅助观测指标：**

| 指标 | 定义 | 说明 |
|---|---|---|
| 影响分析覆盖率 | 模型变更前能自动给出影响范围的比例 | 验证依赖图谱和影响分析机制是否生效 |
| 主线完整度 | 实例数据具备完整 TraceLink 链路的比例 | 验证 TraceLink 自动生成机制是否生效 |
| ITP 安装预检通过率 | ITP 首次安装时预检（版本、依赖、冲突、合规）一次通过的比例 | 验证 ITP 包治理机制的成熟度 |
| 能力模块沉淀数 | 从标杆域项目中提炼为可复用能力模块的数量 | 验证领域功能是否真正从项目沉淀为产品资产 |

- **度量节奏建议：**
  - **T0 基线采集：** 选定风洞试验标杆项目，记录旧模式下的交付周期、人工搬运工作量和升级排查时间
  - **T1 试点验证：** 首个 ITP 闭环后，对照核心指标和辅助指标，校准具体目标数字
  - **T2 推广校准：** 扩展到第 2-3 个同域项目时，校准复用率和差量占比的目标区间
  - **T3 跨域验证：** 扩展到 CDM / MIS 时，补充编排和规则维度的度量指标
