## Slide 1：执行摘要——这次专题汇报要拍什么板

- **页面目标：** 让领导和各类专家在第一页就知道结论、价值和决策点
- **视觉建议：** 四宫格，分别对应“为什么现在做 / 已形成什么共识 / 预期价值 / 需要拍什么板”
- **页面核心内容：**
  - **为什么现在必须做：**
    - 现有底座具备配置工具，但缺少可沉淀、可安装、可治理的业务资产载体
    - 项目交付仍然高度依赖公司侧按项目单独建模开发，现场再导入、调整、验证，导致复用弱、演进难、升级风险高
  - **这次已形成的核心共识：**
    - 以 `ITP` 作为模型化资产载体
    - 采用 `M0 / M0.5 / M1 / M2` 模型继承轴
    - 实例数据不进入 `M` 轴，运行时绑定 `schemaVersionRef`
    - 运行时能力收敛为 `Object Model / Task Orchestration / Workflow / Toolflow`，由 `Trace Service` 提供横切协同
  - **预期价值：**
    - 交付方式从“公司侧按项目建模开发 + 现场导入调整”转向“标准 ITP 导入 + 受控差量”
    - 领域专有功能从“沉在单项目方案里”转向“沉淀为可复用的产品资产和能力模块”
    - 在统一底座上通过 ITP 组合和差量扩展，更快覆盖相邻行业场景
    - 客户差异化从“改坏标准包”转向“标准包之上的受控差量”
    - 模型、依赖、主线和影响分析进入统一治理闭环
  - **本次希望拍板事项：**
    - 是否认可“模型化”作为底座 `v0.5` 的核心架构方向
    - 是否认可 `M` 轴、实例边界与运行时职责边界
    - 是否同意按“双线推进”方式落地：底座修路 + 首个标杆 `ITP` 试点
- **建议口播：**
  > 今天我们希望达成的，不是某一个小功能点的评审，而是底座演进方向的拍板。核心结论是：模型化不是给现有建模器换个名字，而是把业务概念上升为一等架构资产，并通过 ITP、M 轴和运行时治理体系，把交付方式从“工具交付”改造成“资产交付”。

## Slide 2：问题不在工具不够，而在资产没有沉淀

- **页面目标：** 把原来三页痛点压缩成一页，避免铺垫过长
- **视觉建议：** 左侧“旧模式工作流”，右侧“结果问题”，中间用红色箭头标出断裂点
- **页面核心内容：**
  - **现状：**
    - 有表单、流程、数据建模工具
    - 也支持导入导出、规则配置和二次扩展
  - **真正的问题：**
    - 输出的是零散配置，不是可安装的业务资产
    - 公司侧仍需按项目单独建模开发、跨模块拼装、写胶水逻辑，现场再继续导入、调整和验证
    - 导出的多是 SQL、XML、碎片配置，缺少业务上下文、依赖关系和治理规则
  - **直接后果：**
    - 复用困难：换一个客户，核心工作大多重来
    - 资产流失：标杆项目经验难以沉淀为标准包
    - 产品化不足：领域专有功能沉在项目方案里，难以变成标准产品能力
    - 行业扩展慢：进入相邻行业或新场景时，仍需从单项目方案重新改造
    - 升级高风险：平台升级容易牵动客户个性化配置
    - 治理薄弱：缺少影响分析、依赖图谱和可追溯主线
- **一句话结论：**
  - **我们不是缺建模工具，而是缺“业务资产的正式载体”和“围绕资产运行的治理机制”。**
- **建议口播：**
  > 过去我们一直在增强工具能力，但真正的断点不在工具本身，而在于这些工具产出的内容并没有被收敛成一个完整、可复用、可治理的业务资产。因此每个项目看似都在复用配置，实质上仍然是公司侧为单个项目生产一次性方案，再到现场做导入、适配和验证。

## Slide 3：什么是模型化——把业务概念提升为一等架构资产

- **页面目标：** 定义模型化，并引出 `ITP`
- **视觉建议：** 左侧“碎片能力”，右侧“ITP 资产包”，中间用“升维”箭头连接
- **页面核心内容：**
  - **核心定义：**
    - **模型化 = 把业务概念从硬编码和零散配置，提升为平台可注册、可版本化、可自动联动、可差异化扩展、可标准化交换的一等架构资产。**
  - **模型化的资产载体：`ITP`（Industry Template Package）**
    - `Schema`：业务对象、关系、数据组织方式
    - `Templates`：活动模板、生命周期、管控流程、事件联动
    - `Behavior`：视图、页面、扩展挂接
    - `Runtime`：主数据要求、连接器契约、合规规则、种子数据
  - **它带来的交付方式变化：**
    - 从“公司侧按项目建模开发 + 现场导入调整”转向“底座 + 标准 ITP 导入 + 受控微调”
    - 从“项目代码资产”转向“领域业务资产”
    - 从“领域功能沉在项目里”转向“领域功能沉淀为产品能力，并通过 ITP 快速扩展行业应用”
- **建议口播：**
  > 模型化不是把旧有低代码体系重新命名，而是把原本散落在表结构、流程定义、脚本和页面配置里的业务知识，重新组织为可注册、可装载、可治理的正式资产。这份资产的载体，就是 ITP。更重要的是，过去沉在单项目里的领域专有功能，可以开始沉淀为标准产品能力，再通过 ITP 组合和差量扩展，更快覆盖新的行业场景。

  > 如果前面三页还停留在概念层，这一页就把 ITP 具像化。它是一个有 ID、有版本、有兼容边界、可安装、可校验的正式资产包。它里面装的不是单一对象模型，而是一整套业务机制。更重要的是，这些机制可以按能力模块沉淀下来，成为今后扩展相邻行业场景的标准装配单元。客户个性化诉求也不是直接去改标准包，而是在标准包之上叠加 M2 差量。

  - 导入后不是空平台，而是直接形成业务工作台

  - `Workflow` 只管审批子流程，`Toolflow` 只管工具执行
  - `TraceLink` 自动把需求、试验、数据、报告串起来

  > 这页展示的不是最终产品 UI，而是模型化导入后的运行形态。实施现场不再只是导入一批静态配置后再持续调改，而是直接得到一个可运行的业务工作台。业务导航、对象关系、活动模板、审批挂接和工具执行都已经成型。客户特有字段和规则以 M2 差量叠加。底部的主线也不是后补，而是随运行自动形成。更关键的是，这个工作台不是某个项目独有页面，而是领域产品能力的运行形态，后续可以通过替换或组合 ITP 快速扩展到相邻行业场景。

## Slide 6：关键架构决策一——`M` 轴分层与实例数据边界

- **页面目标：** 解决“标准化”和“客户差异化”如何并存的问题，并主动澄清实例边界
- **视觉建议：** 四层结构图 + 独立实例层，明确“实例数据不在继承轴”
- **页面核心内容：**
  - **四级模型继承轴：**
    - `M0`：平台内核模型
    - `M0.5`：公共业务对象
    - `M1`：领域业务模型 / 标准 ITP
    - `M2`：客户差异化模型
  - **关键边界：**
    - 实例数据不属于 `M` 轴
    - 实例对象在创建、导入、重大迁移时绑定 `schemaVersionRef`
    - 模型升级、兼容性校验、影响分析和回溯视图基于实例绑定的模型版本解释
  - **结论：**
    - 标准化与差异化不再相互污染
    - 平台升级不再默认等于客户实例自动回滚
    - 兼容性变化需要迁移计划和兼容性报告
- **建议口播：**
  > 这一页要回答两个问题。第一，标准包和客户个性化如何共存。答案是通过 M0 到 M2 的继承和差量叠加。第二，实例数据怎么办。答案是实例数据不进入 M 轴，它们是运行时对象，必须绑定具体版本，不能把“模型回滚”误解成“实例回滚”。

## Slide 7：关键架构决策二——运行时服务边界与状态归属

- **页面目标：** 修正“谁持有业务真相状态”的常见误解
- **视觉建议：** 四服务协同图，突出“谁管什么状态、谁不管什么状态”
- **页面核心内容：**
  - **`L3` 领域聚合：**
    - 持有业务对象真相状态
    - 如 `Requirement / WorkPackage / Test / Contract`
  - **`Task Orchestration`：**
    - 持有活动实例状态、生命周期阶段状态、事件联动执行状态
    - 解释并执行活动模板、阶段模板和事件链声明
  - **`Workflow Service / Activiti`：**
    - 只持有人工审批子流程状态
  - **`Toolflow Orchestration`：**
    - 只持有工具执行运行时状态
  - **`Trace Service`：**
    - 作为横切能力，承接 `TraceIntent`，生成 `TraceLink`
  - **一句话结论：**
    - **业务真相不交给编排层托管，编排层只管理编排实例状态。**
- **建议口播：**
  > 这页是整场汇报里最容易被技术专家追问的地方。我们这次不是再造一个万能状态机，而是把状态边界理顺。L3 领域聚合持有业务真相，Task 编排层只管理编排实例，Workflow 只做审批，Toolflow 只做工具执行，Trace 负责横切追踪。这种收敛方式，才真正有利于模型化落地后的稳定运行。

## Slide 8：关键架构决策三——治理闭环：依赖图谱、影响分析与 TraceLink

- **页面目标：** 把“高级能力”讲成“模型化成立后的自然治理结果”
- **视觉建议：** 左边模型依赖图谱，右边实例主线图谱，中间用统一 Graph Service 串联
- **页面核心内容：**
  - **模型级治理：依赖图谱 + 影响分析**
    - `ITP` 装载时自动扫描 `schema / templates / behavior / runtime / semantic-refs`
    - 构建模型级依赖图谱
    - 删除、修改、裁剪前给出直接影响、间接影响、阻断或降级建议
  - **实例级治理：TraceLink 主线**
    - 模型定义关系类型
    - 运行时对象产生 `TraceIntent`
    - `Trace Service` 幂等生成或更新 `TraceLink`
    - 主线服务基于 `TraceLink` 计算正向追踪、反向溯源和变更影响
  - **统一观点：**
    - 模型级依赖图谱和实例级主线图谱，本质上都是“图上的关系管理和遍历”
    - 可由同一个图服务体系支撑
- **建议口播：**
  > 影响分析和 TraceLink 不是后加的亮点功能，而是模型化以后必然出现的治理闭环。前者解决“改之前是否知道会影响什么”，后者解决“运行之后能否追踪、溯源和分析影响”。一个面向模型定义，一个面向运行实例，本质上都是图关系管理问题。

  - **仍需继续验证或细化：**

    - ITP 加载、卸载、模块级回滚机制
    - `Task / Workflow / Toolflow / Trace` 的详细时序与回调点
  - **一句话结论：**
    - **方向和主边界已经明确，接下来要进入的是原型验证和细节收敛，而不是重新回到方向争论。**
- **建议口播：**
  > 今天不是来宣称所有细节都已经完成，而是说明哪些关键边界已经达成共识、可以进入蓝图 v0.5，哪些部分需要继续通过 POC 和细化设计推进。这样既有利于领导拍板，也有利于专家形成清晰预期。

## Slide 9.5：首个标杆域比选——为什么建议从 TDM 切入

- **页面目标：** 回答"为什么选 TDM 而不是 CDM 或 MIS 先打样"，主动封堵现场质疑
- **视觉建议：** 三列雷达图 + 下方总结栏，雷达维度统一，颜色区分三域
- **页面核心内容：**
  - **比选维度与评分：**

    | 评估维度 | TDM（试验数据管理） | CDM（协同研发管理） | MIS（数字化运营） | 说明 |
    |---|---|---|---|---|
    | 对象模型复杂度 | 5 | 3 | 2 | 越复杂越能验证 Schema / ITP 承载力 |
    | 编排逻辑复杂度 | 2 | 5 | 4 | CDM/MIS 编排重，但 Task Orchestration 尚未收敛定稿 |
    | 外部集成依赖 | 2 | 5 | 3 | CDM 连接器密集，首轮试点风险高 |
    | 规则个性化程度 | 2 | 3 | 5 | MIS 规则极度客户化，M2 差量机制尚未闭环验证 |
    | 主线可验证性 | 5 | 3 | 2 | TDM 天然具备 Requirement - Test - DataSet - Report 完整主线 |
    | 已有项目积累 | 4 | 3 | 3 | 风洞场景已有标杆项目，可直接提炼标准 ITP |
    | 团队就绪度 | 4 | 3 | 3 | 行业团队对 TDM 领域理解最深 |

  - **关键判断逻辑：**
    - **TDM 的核心优势：** 对象模型最丰富（schema 填充最重），天然具备完整数据主线，可以一次性验证 ITP 包结构、M 轴继承、Schema 注册、TraceLink 自动生成和影响分析等核心机制
    - **CDM 为什么不宜首选：** 编排逻辑和连接器依赖最重，而 Task Orchestration 的活动模板 schema 和详细时序仍在待细化清单中，首轮贸然切入容易卡在底座能力未就绪
    - **MIS 为什么不宜首选：** 规则极度个性化，M2 差量和合规约束机制是其核心价值，但合规声明 schema 和兼容性引擎尚未定稿，先打容易暴露过多未收敛问题
    - **后续扩展路径：** TDM 验证闭环后 - CDM 扩展编排和连接器能力 - MIS 扩展规则和合规能力
  - **一句话结论：**
    - **TDM 是"模型最重、编排最轻、主线最完整"的域，最适合作为首个标杆域验证 ITP 核心架构机制，同时具备从已有项目中提炼标准 ITP 的现实条件。**
- **建议口播：**
  > 为什么建议率先在 TDM 切入？核心原因有三个。第一，TDM 的对象模型最丰富，最能检验 ITP 包结构和 Schema 注册能力。第二，TDM 天然具备从需求到报告的完整数据主线，可以一次性验证 TraceLink 和影响分析。第三，TDM 对编排和外部集成的依赖最轻，不会因为底座尚未收敛的编排能力而卡住试点进度。CDM 和 MIS 则分别在编排和规则维度更重，更适合在底座侧相应能力就绪后再逐步扩展。

### Appendix F：成功度量指标框架

- **用途：** 为模型化架构落地提供可量化的验证基线和目标值，供标杆域试点阶段和后续推广阶段对照使用
- **度量设计原则：** 首轮试点以方向性区间为目标，不预设绝对数字；待首个 ITP 闭环后，根据实测数据校准目标值

| 指标 | 定义 | 旧模式基线（估） | 首个 ITP 试点目标 | 度量方式 |
|---|---|---|---|---|
| **交付周期** | 从需求确认到客户可运行环境交付的日历时间 | 月级（通常 2-4 个月） | 周级（目标 2-4 周） | 项目里程碑记录 |
| **复用率** | 标准 ITP 内的 Schema / Template / Behavior 被后续项目直接复用（含 M2 差量叠加）的比例 | < 20%（配置碎片手工搬运） | 大于等于 60%（首批 2-3 个同域项目） | ITP 安装日志 + 差量统计 |
| **升级成功率** | 底座或 ITP 版本升级后，客户环境一次性通过兼容性校验并正常运行的比例 | 低（升级常需现场逐项排查） | 大于等于 80%（基于 SchemaVersion 兼容性引擎 + 预检报告） | 升级预检报告 + 回归验证结果 |
| **客户差量占比** | M2 差量（客户个性化）占整个 ITP 总资产的比例 | 无法度量（标准与个性化混杂） | 小于等于 30%（标准包大于等于 70%） | ITP manifest 中 M2 overrides 统计 |

- **辅助观测指标：**

| 指标 | 定义 | 说明 |
|---|---|---|
| 影响分析覆盖率 | 模型变更前能自动给出影响范围的比例 | 验证依赖图谱和影响分析机制是否生效 |
| 主线完整度 | 实例数据具备完整 TraceLink 链路的比例 | 验证 TraceLink 自动生成机制是否生效 |
| ITP 安装预检通过率 | ITP 首次安装时预检（版本、依赖、冲突、合规）一次通过的比例 | 验证 ITP 包治理机制的成熟度 |
| 能力模块沉淀数 | 从标杆域项目中提炼为可复用能力模块（Level 3）的数量 | 验证领域功能是否真正从项目沉淀为产品资产 |

- **度量节奏建议：**
  - **T0 基线采集：** 选定风洞试验标杆项目，记录旧模式下的交付周期、人工搬运工作量和升级排查时间
  - **T1 试点验证：** 首个 ITP 闭环后，对照 4 项核心指标和 4 项辅助指标
  - **T2 推广校准：** 扩展到第 2-3 个同域项目时，校准复用率和差量占比的目标区间
  - **T3 跨域验证：** 扩展到 CDM / MIS 时，补充编排和规则维度的度量指标
